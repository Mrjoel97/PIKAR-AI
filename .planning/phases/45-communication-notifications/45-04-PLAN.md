Implementation:
1. Guard: `if provider not in _NOTIF_PROVIDERS: raise HTTPException(400, "Not a notification provider")`
2. Lazy-import `dispatch_notification` from `app.services.notification_dispatcher`
3. Call `result = await dispatch_notification(user_id=current_user_id, event_type="agent.message", payload={"text": "This is a test notification from Pikar. If you see this message, your notification channel is working correctly."})`
4. Check if the specific provider succeeded: `sent = result.get(provider, False)`
5. If `sent` is False, return `JSONResponse(status_code=502, content={"detail": f"Failed to deliver test notification via {provider}. Check that the provider is connected and a notification rule exists for the 'agent.message' event type."})`. The 502 tells the frontend the downstream delivery failed, not that the endpoint itself is broken.
6. If `sent` is True, return `JSONResponse(content={"provider": provider, "sent": True})`

Add a Google-style docstring:
```
"""Send a test notification to verify provider connectivity.

Dispatches a sample ``agent.message`` event through the notification
pipeline.  The user must have at least one enabled notification rule
for the ``agent.message`` event type (or a rule that matches this
event) for the message to be delivered.

Args:
    provider: Notification provider key (``"slack"`` or ``"teams"``).
    current_user_id: Authenticated user's UUID.

Returns:
    JSON confirming delivery success.

Raises:
    HTTPException: 400 if provider is not a notification provider.

"""
```

Important: The `dispatch_notification` function handles the full fan-out including looking up notification rules, resolving provider credentials, and calling the appropriate service (SlackNotificationService or TeamsNotificationService). The endpoint does NOT need to import or call those services directly.

Note: If the user has no `agent.message` notification rule configured, `dispatch_notification` will match zero rules and the result dict will be empty (no provider key present). In that case `result.get(provider, False)` returns False, and the 502 response message guides the user to create a rule. This is correct behavior — the test button validates the full pipeline including rule matching.
