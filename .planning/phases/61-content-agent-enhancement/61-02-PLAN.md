<objective>
Add auto-schedule suggestion after content creation so the Content Agent recommends optimal posting times and the user can confirm to auto-schedule via the content calendar.

Purpose: Currently content creation and scheduling are disconnected -- the user creates content and must separately think about when to post. This plan closes that gap by having the agent proactively suggest timing and offer one-click scheduling.

Output: New `suggest_and_schedule_content` tool, updated agent instructions, unit tests.
</objective>

    - Test 6: Content type mapping: "social_post" maps to content_calendar type "social", "blog_intro" maps to "blog", "email" maps to "email"
  </behavior>
  <action>
    Add `suggest_and_schedule_content` async tool function to `app/agents/content/tools.py`. The tool should:

    1. Accept parameters:
       - `title: str` -- content title
       - `content_type: str` -- the type of content (social_post, blog_intro, email, video, etc.)
       - `platform: str | None = None` -- target platform
       - `description: str = ""` -- brief description of the content
       - `schedule: bool = False` -- if True, actually schedule it; if False, just suggest
       - `preferred_date: str | None = None` -- user override for date (YYYY-MM-DD)
       - `preferred_time: str | None = None` -- user override for time (HH:MM)

    2. **Compute optimal timing:**
       - Import `PLATFORM_GUIDELINES` from `app.agents.tools.publishing_strategy`
       - Look up the platform in PLATFORM_GUIDELINES to get optimal_times string
       - Parse the first recommended time slot into a concrete date/time:
         - Use `datetime` to find the next occurrence of a recommended day (e.g., "Tue-Thu" -> next Tuesday)
         - Pick the first time from the list (e.g., "8am-10am" -> "09:00")
       - If `preferred_date` or `preferred_time` are provided, use those as overrides
       - If platform is None or not in guidelines, default to "next weekday at 10:00"
       - Build reasoning string explaining why this time was chosen (e.g., "LinkedIn posts perform best Tue-Thu 8-10am in your audience's timezone")

    3. **Map content_type to calendar content_type:** Use a mapping dict: `{"social_post": "social", "blog_intro": "blog", "email": "email", "caption": "social", "headline": "ad", "tagline": "ad", "video": "video"}`. Default to "other".

    4. **If schedule=False** (suggestion mode): Return `{"success": True, "mode": "suggestion", "optimal_date": date_str, "optimal_time": time_str, "reasoning": reasoning, "platform": platform, "content_type": mapped_type, "message": "I suggest posting on {date} at {time}. {reasoning}. Say 'schedule it' to confirm."}`.

    5. **If schedule=True** (scheduling mode): Call `ContentCalendarService().schedule_content(title=title, content_type=mapped_type, scheduled_date=date_str, platform=platform, scheduled_time=time_str, description=description, user_id=get_current_user_id())`. Return `{"success": True, "mode": "scheduled", "calendar_item": result, "message": "Scheduled '{title}' for {date} at {time} on {platform}."}`.

    For tests, mock `ContentCalendarService` and `datetime` where needed. Use `pytest.mark.asyncio`.
