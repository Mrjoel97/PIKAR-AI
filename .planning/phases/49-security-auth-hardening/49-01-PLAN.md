<objective>
Add a server-side Next.js 16 root proxy (`frontend/proxy.ts`) that intercepts every protected route, verifies the Supabase auth cookie via `getClaims()`, refreshes expired tokens, and redirects unauthenticated requests to `/auth/login` BEFORE any page HTML or RSC payload is streamed to the browser.

Purpose: AUTH-01 closes the largest production-readiness security gap: today the app uses only client-side `ProtectedRoute.tsx` which renders page content first and then redirects on the client — meaning unauthenticated visitors briefly see protected page shells, prefetch protected RSC payloads, and (more critically) any page that doesn't import `ProtectedRoute` is fully exposed.

<critical_decisions>
1. **File name: `proxy.ts` not `middleware.ts`** — Next.js 16 renamed this convention.
   Source: Next.js 16 docs (verified during planning research).
   Place at `frontend/proxy.ts` (project root, sibling to package.json).

2. **Use `getClaims()` not `getSession()`** — `getSession()` trusts the cookie, which can be
   spoofed in server contexts. `getClaims()` validates the JWT signature on every call.
   Source: Supabase SSR docs (verified during planning research).

3. **Protected path list must EXACTLY match the success criteria:**
   - `/dashboard` and all `/dashboard/*` (including `/dashboard/team`, `/dashboard/finance`, etc.)
   - `/settings` and all `/settings/*`
   - `/admin` and all `/admin/*`
   - All `(personas)` routes — `/solopreneur/*`, `/startup/*`, `/sme/*`, `/enterprise/*`
   - `/onboarding/*`, `/approval/*`, `/departments/*`, `/org-chart/*`

4. **Unprotected paths (matcher must NOT redirect):**
   - `/` (landing page)
   - `/auth/*` (login, signup, callback)
   - `/api/*` (API routes — backend handles its own auth)
   - `/_next/*` (Next.js internals)
   - `/favicon.ico`, `/robots.txt`, image/font assets

5. **Token refresh contract:** When the cookie is present but the access token is expired, the
   proxy must call `getClaims()` (which auto-refreshes via the refresh_token cookie) and
   forward the refreshed cookies to the response so the browser receives the new tokens.

6. **NextResponse pattern:** Build a single `response = NextResponse.next({ request })` at the
   top of the proxy, hand its cookies object to the Supabase SSR client, then RETURN that same
   response object. Creating new responses mid-flow drops the refreshed cookies.
</critical_decisions>

    ```typescript
    // frontend/src/lib/supabase/proxy.ts
    import { createServerClient } from '@supabase/ssr';
    import { NextRequest, NextResponse } from 'next/server';

    type ClaimsResult = { sub: string; email?: string } | null;

    /**
     * Refresh the Supabase session for an incoming request and return both the
     * outgoing response (with any refreshed cookies) and the validated JWT claims.
     *
     * Uses getClaims() — NOT getSession() — because getSession() trusts the raw
     * cookie which can be spoofed in server-side contexts. getClaims() validates
     * the JWT signature against published JWKS on every call.
     */
    export async function updateSession(
      request: NextRequest,
    ): Promise<{ response: NextResponse; claims: ClaimsResult }> {
      // CRITICAL: a single response object MUST be reused throughout this function
      // so refreshed Set-Cookie headers are not dropped.
      let response = NextResponse.next({ request });

      const supabase = createServerClient(
        process.env.NEXT_PUBLIC_SUPABASE_URL!,
        process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
        {
          cookies: {
            getAll() {
              return request.cookies.getAll();
            },
            setAll(cookiesToSet) {
              // Mirror the new cookies to BOTH the request (for downstream RSCs)
              // and the response (for the browser).
              for (const { name, value } of cookiesToSet) {
                request.cookies.set(name, value);
              }
              response = NextResponse.next({ request });
              for (const { name, value, options } of cookiesToSet) {
                response.cookies.set(name, value, options);
              }
            },
          },
        },
      );

      const { data, error } = await supabase.auth.getClaims();
      if (error || !data?.claims) {
        return { response, claims: null };
      }
      return {
        response,
        claims: { sub: data.claims.sub, email: data.claims.email as string | undefined },
      };
    }
    ```

    ```typescript
    // frontend/proxy.ts
    import { NextRequest, NextResponse } from 'next/server';
    import { updateSession } from '@/lib/supabase/proxy';

    // Path prefixes that REQUIRE an authenticated session.
    const PROTECTED_PREFIXES = [
      '/dashboard',
      '/settings',
      '/admin',
      '/onboarding',
      '/approval',
      '/departments',
      '/org-chart',
      '/solopreneur',
      '/startup',
      '/sme',
      '/enterprise',
    ];

    function isProtected(pathname: string): boolean {
      return PROTECTED_PREFIXES.some(
        (prefix) => pathname === prefix || pathname.startsWith(`${prefix}/`),
      );
    }

    export async function proxy(request: NextRequest): Promise<NextResponse> {
      const { pathname } = request.nextUrl;

      // Always refresh the session so RSCs downstream see fresh tokens, even on
      // public pages. Only enforce the redirect on protected paths.
      const { response, claims } = await updateSession(request);

      if (isProtected(pathname) && !claims) {
        const loginUrl = new URL('/auth/login', request.url);
        loginUrl.searchParams.set('next', pathname);
        return NextResponse.redirect(loginUrl);
      }

      return response;
    }

    export const config = {
      // Run on every request EXCEPT static assets, images, favicon, and api routes.
      // (api routes have their own backend auth via verify_token.)
      matcher: [
        '/((?!api|_next/static|_next/image|favicon.ico|robots.txt|.*\\.(?:png|jpg|jpeg|gif|svg|webp|woff|woff2|ttf|ico)$).*)',
      ],
    };
    ```
