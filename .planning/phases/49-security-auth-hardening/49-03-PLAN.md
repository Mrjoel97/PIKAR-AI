<objective>
Reconcile, verify, and lock the workspace RBAC role assignment UI to ship the canonical
three-role system (admin / editor / viewer) end-to-end. The infrastructure already exists
(Phase 35 shipped `workspace_members`, `WorkspaceService.update_member_role`,
`PATCH /teams/members/{member_user_id}/role`, `RoleDropdown.tsx`, `TeamMemberList.tsx`,
`/dashboard/team/page.tsx`), but it has never been verified end-to-end against AUTH-03's
success criteria, the existing UI lives behind the `teams` feature gate (which restricts it
to startup tier and above), and the role labels need final reconciliation against the
ROADMAP wording ("admin, member, or viewer").

Purpose: AUTH-03 requires that "a workspace admin can assign admin, member, or viewer roles
to other users via the RBAC interface." The schema, service, and UI all already use the
three-role taxonomy (admin/editor/viewer) — the ROADMAP wording "member" is a synonym for
"editor". This plan locks the wording, surfaces the role-management UI to ALL workspace
admins (not just teams-feature-gated tiers), and adds backend test coverage proving the
role assignment endpoint actually persists changes and rejects invalid actors.

Output:
- Backend tests proving `WorkspaceService.update_member_role` enforces admin-only +
  validates role values + persists correctly (covers AUTH-03 success criterion #3)
- Frontend dropdown UI verified end-to-end with the three canonical roles
- Removal of the `teams` feature gate from the read-only team page so any workspace admin
  (not just startup-tier+) can see and use the RBAC interface
</objective>

The ROADMAP says "admin, member, or viewer" but the codebase says "editor" instead of "member".
DECISION: keep "editor" as the schema/code identifier (it is already shipped end-to-end)
and surface "Editor" as the user-facing label. Document this in the SUMMARY so anyone
reading the ROADMAP knows "member" === "editor".

Existing endpoint signature in app/routers/teams.py:
```python
@router.patch("/members/{member_user_id}/role")
async def update_member_role(
    request: Request,
    member_user_id: str,
    body: UpdateRoleRequest,
    user_id: str = Depends(get_current_user_id),
    _admin: None = Depends(require_role("admin")),
):
    """Update a workspace member's role. Admin only."""
```

The endpoint is already gated by `require_role("admin")` middleware (verified at
app/middleware/workspace_role.py). The service `WorkspaceService.update_member_role`
also validates the actor is an admin and prevents owner role changes.

Existing TeamMemberList uses fetchWithAuth from frontend/src/services/api.ts:
```ts
await fetchWithAuth(`/teams/members/${memberId}/role`, { method: 'PATCH', body: JSON.stringify({ role: newRole }) });
```

The `teams` feature gate is currently applied at the router level:
```python
router = APIRouter(
    prefix="/teams",
    tags=["Teams"],
    dependencies=[Depends(require_feature("teams"))],
)
```
This restricts BOTH read AND write to startup tier+. AUTH-03 requires that a workspace admin
on ANY tier can assign roles, so the role-management endpoints (specifically
`PATCH /members/{id}/role`) must move out from under the blanket `teams` gate. Solo users
without a workspace are no-ops (require_role short-circuits).
</interfaces>

<critical_decisions>
1. **Lock "editor" as the schema identifier, "Editor" as the UI label** — do NOT rename
   the schema column or migrate data. The ROADMAP wording "member" is a synonym; the
   SUMMARY must document this explicitly.

2. **Three-role system stays at admin/editor/viewer** — no new "member" tier.

3. **Decouple the role-management endpoint from the `teams` feature gate** — the
   `require_feature("teams")` router-level dependency restricts to startup-tier+ users.
   AUTH-03 must work for any workspace admin. SOLUTION: split the router so
   `PATCH /teams/members/{user_id}/role` and `GET /teams/me` (workspace context) are
   always available, while team analytics endpoints stay behind the `teams` gate.

   IMPLEMENTATION: do NOT restructure the entire router (too risky). Instead, add the
   per-endpoint override pattern: re-declare the role-management endpoints in a sibling
   sub-router without the feature gate, OR remove `require_feature("teams")` from
   `update_member_role` and `get_workspace_me` specifically.

   CHOSEN PATH: keep the router-level gate but add `dependencies=[]` override on the
   role-management endpoints by extracting them into a separate sub-router under the
   same `/teams` prefix WITHOUT the feature gate. This is ~15 lines of router code.

4. **Solo users have no workspace** — `require_role("admin")` short-circuits with a
   pass-through when `service.get_workspace_for_user()` returns None. This is already
   the behavior in app/middleware/workspace_role.py — verify it stays.

5. **Owner role is immutable** — the existing service raises ValueError when attempting
   to change the owner's role. Backend test must cover this.

6. **No new database migration** — schema is already correct. This plan only adds tests
   and removes/relocates one feature-gate dependency.
</critical_decisions>
