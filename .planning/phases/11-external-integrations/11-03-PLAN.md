  <action>
    Create `frontend/src/app/(admin)/integrations/page.tsx` following the analytics page pattern.

    **Page structure:**

    `'use client'` directive at top. Imports: `useCallback, useEffect, useState` from React, `createClient` from `@/lib/supabase/client`, `ProviderCard` and `ConfigureModal` from the components created in Task 1.

    **TypeScript interface:**
    ```typescript
    interface IntegrationStatus {
      provider: string;
      is_active: boolean;
      health_status: string;
      key_last4: string | null;
      base_url: string | null;
      config: Record<string, string>;
      updated_at: string;
    }
    ```

    **State:**
    - `integrations: IntegrationStatus[]` — loaded from API
    - `isLoading: boolean` — initial load state
    - `fetchError: string | null` — error message
    - `configuringProvider: string | null` — which provider's modal is open (null = closed)
    - `isSaving: boolean` — modal save loading state
    - `testingProvider: string | null` — which provider is being tested (null = none)
    - `testResult: { provider: string; healthy: boolean; error?: string } | null` — last test result (briefly shown)

    **API calls:**

    A) `fetchIntegrations` (useCallback, called on mount):
       - Get session via `supabase.auth.getSession()`
       - `GET ${API_URL}/admin/integrations` with Bearer token
       - If response is empty array, show all 4 providers as unconfigured cards (generate placeholder entries for missing providers)
       - The API returns only configured providers. For UI, always show all 4 providers. Merge API response with the full provider list: `["sentry", "posthog", "github", "stripe"]`. For providers not in API response, create default entries with `is_active: false, health_status: "unknown", key_last4: null, config: {}`.

    B) `handleSave` (called by ConfigureModal onSave):
       - `PUT ${API_URL}/admin/integrations/${configuringProvider}` with Bearer token
       - Body: `{ api_key, base_url, config }` from modal
       - On success: close modal, refetch integrations
       - On error: show error (could use window.alert for MVP, matching existing patterns)

    C) `handleTestConnection` (called by ProviderCard onTestConnection):
       - Set `testingProvider` to the provider name
       - `POST ${API_URL}/admin/integrations/${provider}/test` with Bearer token
       - Set `testResult` with the response
       - Clear `testingProvider`
       - After 5 seconds, clear `testResult` (auto-dismiss)
       - On success: refetch integrations (health_status may have updated)

    D) `handleDelete` — not in UI for MVP (disconnect is done by deleting the key via configure modal or future iteration). Keep the code path available but do not render a delete button.

    **Rendering:**

    - Page header: `<h1>` "Integrations" with subtitle "Connect external services to monitor and manage your platform"
    - If test result exists, show a toast-like banner at the top: green "Connection successful" or red "Connection failed: {error}"
    - 2x2 CSS Grid of ProviderCards (`grid grid-cols-1 md:grid-cols-2 gap-6`)
    - Each card passes: provider data from state, onConfigure handler (sets configuringProvider), onTestConnection handler, isTesting state
    - ConfigureModal rendered when configuringProvider is not null, with currentConfig from the matching integration entry

    **No polling needed** — integrations don't change frequently, only on user action. Refetch after save and test-connection is sufficient.

    Tailwind classes: page padding `p-8`, heading `text-2xl font-bold text-white`, subtitle `text-gray-400 mt-1 mb-8`.
