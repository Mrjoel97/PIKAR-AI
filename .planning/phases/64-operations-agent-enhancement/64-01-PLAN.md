    - Test: analyze_bottlenecks returns step-level stats (avg_duration_hours, max_duration_hours, execution_count, failure_rate) grouped by step_name
    - Test: steps with avg_duration_hours > 24 are flagged as bottlenecks
    - Test: steps with failure_rate > 0.2 (20%) are flagged as bottlenecks
    - Test: steps with status="waiting_approval" for avg > 48 hours are flagged as approval bottlenecks
    - Test: recommendations are generated in plain English (e.g., "Content Approval averages 3.2 days -- consider setting up approver reminders")
    - Test: analyze_bottlenecks returns empty results gracefully when user has no workflow executions
    - Test: get_workflow_health_summary returns overall completion rate, avg execution time, and top 3 bottlenecks
  </behavior>
  <action>
    1. Create `app/services/workflow_bottleneck_service.py`:
       - Class `WorkflowBottleneckService` extending `BaseService`.

       - Method `async analyze_bottlenecks(user_id: str, days: int = 30) -> dict`:
         - Query workflow_executions for user_id created within last `days` days.
         - For each execution, query workflow_steps and compute per-step duration:
           `duration_hours = (completed_at - started_at).total_seconds() / 3600` for steps where both timestamps exist.
         - Aggregate across all executions: group by step_name, compute avg_duration_hours, max_duration_hours, count, failure_count (status="failed"), approval_wait_count (status="waiting_approval").
         - Use a single query approach: join workflow_steps with workflow_executions on execution_id, filter by user_id and date range. Use `execute_async` with raw SQL or Supabase PostgREST filters.
         - For efficiency, use a single query:
           ```
           workflow_steps joined to workflow_executions
           WHERE workflow_executions.user_id = :user_id
           AND workflow_executions.created_at > now() - interval ':days days'
           SELECT step_name, status, started_at, completed_at
           ```
         - Then aggregate in Python (Supabase PostgREST cannot do GROUP BY aggregations).

       - Method `_generate_recommendations(step_stats: list[dict]) -> list[dict]`:
         - For each step stat, check thresholds:
           - avg_duration_hours > 24: "**{step_name}** averages {avg:.1f} days -- consider adding reminders or parallel tracks."
           - failure_rate > 0.2: "**{step_name}** fails {pct:.0f}% of the time -- review step configuration or add error handling."
           - approval_wait_rate > 0.3 and avg_duration > 48h: "**{step_name}** waits for approval averaging {avg:.1f} days -- consider auto-approval rules or escalation timers."
           - max_duration_hours > 7 * 24 (1 week): "**{step_name}** has taken up to {max:.0f} days in one case -- investigate outliers."
         - Return list of `{step_name, type: "slow"|"failing"|"approval_blocked"|"outlier", severity: "high"|"medium", message, avg_duration_hours, metric_value}`.
         - Sort by severity (high first), then by avg_duration descending.

       - Method `async get_workflow_health_summary(user_id: str, days: int = 30) -> dict`:
         - Query workflow_executions for user within date range.
         - Compute: total_executions, completed_count, failed_count, completion_rate, avg_execution_hours (from created_at to completed_at for completed ones).
         - Call analyze_bottlenecks and take top 3 recommendations.
         - Return `{total_executions, completion_rate, avg_execution_hours, top_bottlenecks: [...], period_days}`.

       - Module-level convenience: `async def analyze_bottlenecks(user_id, days=30) -> dict`.

    2. Create `tests/unit/services/test_workflow_bottleneck_service.py`:
       - Mock Supabase execute calls returning test data with various step timings.
       - Test analyze_bottlenecks with mix of fast/slow/failing steps.
       - Test _generate_recommendations produces correct message types.
       - Test get_workflow_health_summary aggregates correctly.
       - Test empty execution list returns graceful empty result.
       - Test steps without timestamps (started_at=None) are skipped in duration calc.
  </action>
