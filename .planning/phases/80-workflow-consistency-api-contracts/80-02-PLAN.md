  <action>
**Step 1 -- Install openapi-typescript as a devDependency:**
Run `cd frontend && npm install --save-dev openapi-typescript` from the project root.

**Step 2 -- Create the codegen script** at `frontend/scripts/generate-api-types.mjs`:

This script does NOT require the backend to be running. Instead, it:
1. Uses Python to dump the OpenAPI schema to a JSON file by importing the FastAPI app directly (no server needed)
2. Runs openapi-typescript on that JSON file to produce TypeScript types

The script should use `node:child_process` `execFileSync` (NOT `execSync` with string concatenation) to safely invoke:
- `uv` with args `["run", "python", "-c", "from app.fast_api_app import app; import json; print(json.dumps(app.openapi()))"]` from the project root
- `npx` with args `["openapi-typescript", schemaPath, "-o", outputPath]` from the frontend directory

Use `node:path` resolve for all paths. Write the schema JSON to `frontend/.openapi-schema.json` as an intermediate file, then delete it after codegen. Add `frontend/.openapi-schema.json` to `.gitignore`.

**Step 3 -- Add the script to package.json:**
Add to `"scripts"`: `"generate:types": "node scripts/generate-api-types.mjs"`

**Step 4 -- Run the script to generate the initial types file:**
Run `cd frontend && npm run generate:types` to create `frontend/src/types/api.generated.ts`. Commit this file as the baseline.

**Step 5 -- Add a CI job** to `.github/workflows/ci.yml`:

Add a new job `api-types-check` (parallel with existing jobs, no `needs:`):

```yaml
  api-types-check:
    name: API Types Freshness Check
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install uv
        run: python -m pip install --upgrade uv==0.8.13

      - name: Sync backend dependencies
        run: uv sync --locked

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: npm
          cache-dependency-path: frontend/package-lock.json

      - name: Install frontend dependencies
        working-directory: frontend
        run: npm ci

      - name: Generate API types
        run: cd frontend && npm run generate:types

      - name: Check for type drift
        run: |
          if ! git diff --exit-code frontend/src/types/api.generated.ts; then
            echo "::error::Generated API types are out of date. Run 'cd frontend && npm run generate:types' and commit the result."
            exit 1
          fi
```

**Step 6 -- Add `.openapi-schema.json` to gitignore:**
Append `frontend/.openapi-schema.json` to the project `.gitignore` (or `frontend/.gitignore` if that exists) so the temp schema file is never committed.

<task type="auto">
  <name>Task 2: Replace hand-maintained workflow types with generated imports</name>
  <files>frontend/src/services/workflows.ts</files>
  <action>
After the generated types file exists, update `frontend/src/services/workflows.ts` to import from it instead of defining types inline.

**Step 1 -- Identify which generated type names match:**

The generated types file uses OpenAPI schema component names from FastAPI/Pydantic. Inspect `frontend/src/types/api.generated.ts` to find which `components['schemas']` keys correspond to the hand-maintained types. The names may differ (e.g., the Pydantic model might be `WorkflowExecutionResponse` vs the frontend's `WorkflowExecution`).

**Step 2 -- For types that exist in the generated file:**

Replace inline interface definitions with re-exports from the generated file. Use this pattern:
```typescript
import type { components } from '@/types/api.generated';

// Re-export backend-sourced types for backward compatibility
export type WorkflowExecution = components['schemas']['BackendModelName'];
```

If the generated type name differs from the current frontend name, use type aliasing so all existing imports across the frontend continue to work without changes.

**Step 3 -- For types that do NOT exist in the generated file** (frontend-only types like TimelineStep, ExecutionTimeline, or request types that are not Pydantic models):

Keep them as-is in workflows.ts. These are UI-only types not backed by backend schemas.

**Step 4 -- For types where the generated version is incomplete** (e.g., the backend does not expose all fields the frontend needs):

Use TypeScript intersection types to extend:
```typescript
type BaseExecution = components['schemas']['BackendExecutionModel'];
export interface WorkflowExecution extends BaseExecution {
  template_name?: string;  // frontend-only enrichment from joined query
}
```

**Step 5 -- Verify the frontend still builds:**
Run `cd frontend && npm run build`. If build errors appear due to type mismatches, adjust the type mappings. The goal is zero-breakage migration. If a generated type is missing fields the frontend uses, keep the hand-maintained type but add a `// TODO(ARCH-04): align backend schema to include this field` comment -- do NOT break the build.

**Important:** Only migrate types that have clear 1:1 mappings in the generated file. Focus on `workflows.ts` as the proof-of-concept for this pattern. Other service files (approvals.ts, briefing.ts, initiatives.ts, etc.) can be migrated incrementally in future work.
