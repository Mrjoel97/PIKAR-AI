    - Test 6: Handles case where both variants perform identically (keeps variant A as default)

  <action>
Create `app/services/email_ab_testing_service.py` with class `EmailABTestingService`:

1. Method `async def create_ab_test(self, user_id, sequence_id, step_index, variant_b_subject, variant_b_body, split_pct=50) -> dict`:
   - Fetches the email_sequence_steps for the given sequence
   - Identifies the step at step_index as Variant A (original)
   - Creates a new step row marked as variant_b with `ab_test_id` linking them, `is_variant=True`, `variant_label="B"`, same delay settings
   - Updates the original step with `variant_label="A"` and the same `ab_test_id`
   - `split_pct` controls the traffic split (default 50/50)
   - Returns: `{"ab_test_id": uuid, "variant_a": {...}, "variant_b": {...}, "split_pct": split_pct}`

   Note: The ab_test tracking is stored in the step metadata JSON field (no new migration needed). Fields added to step metadata:
   - `ab_test_id`: UUID linking variants
   - `is_variant`: bool
   - `variant_label`: "A" or "B"
   - `split_pct`: int (0-100, percentage going to this variant)

2. Method `async def get_results(self, user_id, sequence_id, ab_test_id) -> dict`:
   - Queries email_tracking_events for both variant steps
   - Computes per-variant: sends, opens, clicks, open_rate, click_rate
   - Returns: `{"variant_a": {...metrics}, "variant_b": {...metrics}, "winner": "A"|"B"|"insufficient_data", "confidence_note": str}`

3. Method `async def select_winner(self, user_id, sequence_id, ab_test_id, min_sample=50) -> dict`:
   - Calls get_results()
   - If either variant has < min_sample sends, returns `{"winner": "insufficient_data", "reason": "Need at least 50 sends per variant"}`
   - Calculates combined score: 0.7 * open_rate + 0.3 * click_rate
   - Returns winner variant label and scores

4. Method `async def apply_winner(self, user_id, sequence_id, ab_test_id) -> dict`:
   - Calls select_winner()
   - If winner is determined, replaces the original step content with the winner's content
   - Marks the losing variant step as inactive
   - Clears ab_test metadata from the winning step (test complete)
   - Returns confirmation

Uses AdminService for DB access (same pattern as EmailSequenceService). All AB test state is stored in step metadata JSON -- no new database migration required.

Write tests mocking Supabase queries. Verify score calculations, winner selection logic, and edge cases.
