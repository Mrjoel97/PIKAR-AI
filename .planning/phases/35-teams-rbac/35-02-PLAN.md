**A. Create `app/routers/teams.py`:**

A FastAPI router with prefix `/teams`, tags `["Teams"]`, and router-level dependency `Depends(require_feature("teams"))` to ensure only startup+ tiers can access team endpoints.

Define Pydantic request/response models at the top:

```python
class WorkspaceResponse(BaseModel):
    id: str
    name: str
    slug: str | None
    owner_id: str
    role: str  # current user's role in this workspace
    member_count: int

class MemberResponse(BaseModel):
    id: str
    user_id: str
    email: str
    display_name: str | None
    role: str
    joined_at: str

class CreateInviteRequest(BaseModel):
    role: str = Field(default="viewer", pattern="^(editor|viewer)$")
    expires_hours: int = Field(default=168, ge=1, le=720)  # max 30 days

class InviteResponse(BaseModel):
    id: str
    token: str
    role: str
    expires_at: str
    share_url: str  # full URL for sharing

class AcceptInviteRequest(BaseModel):
    token: str

class UpdateRoleRequest(BaseModel):
    role: str = Field(pattern="^(admin|editor|viewer)$")
```

**Endpoints:**

1. `GET /teams/workspace` — Get current user's workspace (calls get_or_create_workspace). Returns WorkspaceResponse. No role gate (any authenticated user on startup+ tier).

2. `GET /teams/members` — List workspace members. Requires any workspace member. Calls get_workspace_members. Returns list[MemberResponse].

3. `POST /teams/invites` — Create invite link. Body: CreateInviteRequest. Requires admin role via `Depends(require_role("admin"))`. Calls create_invite_link. Returns InviteResponse with share_url constructed as `{request.base_url}dashboard/team/join?token={token}`.

4. `POST /teams/invites/accept` — Accept an invite. Body: AcceptInviteRequest. No role gate (user may not be a member yet). Calls accept_invite. Returns membership record. Catch ValueError and return 400.

5. `PATCH /teams/members/{member_user_id}/role` — Update member role. Body: UpdateRoleRequest. Requires admin via `Depends(require_role("admin"))`. Calls update_member_role. Returns updated MemberResponse.

6. `DELETE /teams/members/{member_user_id}` — Remove member. Requires admin via `Depends(require_role("admin"))`. Calls remove_member. Returns `{"success": true}`.

All endpoints use `user_id: str = Depends(get_current_user_id)` for authentication. Add rate limiting with `@limiter.limit(get_user_persona_limit)`. Follow the docstring convention of existing routers (Google-style with short description).

Error handling: Wrap service calls in try/except, return appropriate HTTP status codes (400 for validation errors, 403 for permission errors, 404 for not found, 500 for unexpected errors).

**Provider logic:**

1. On mount, check if user is authenticated (use `createClient()` from `@/lib/supabase/client` to get user)
2. If authenticated, query Supabase directly (matching the pattern used by other contexts in this codebase):
   - Query `workspace_members` joined with `workspaces` where `user_id = currentUser.id`
   - If no row exists, the user has no workspace yet (workspace is created lazily when they visit the team page)
   - Extract workspace_id, role, workspace name, owner_id
   - Count members via a separate count query on workspace_members

3. Derive permission booleans:
   - `canEdit = role === 'admin' || role === 'editor'`
   - `canManageTeam = role === 'admin'`
   - `canView = role !== null`
   - `isOwner = workspace.owner_id === currentUser.id`
   - `isTeamWorkspace = memberCount > 1`

4. Set `ready: true` after initial fetch completes (even if no workspace found — that's a valid state)

5. Provide a `refresh()` function that re-fetches workspace state

**Why this task exists:** TEAM-01 requires that "invited member sees shared initiatives, workflows, and content on their dashboard." Without this, each endpoint filters by `user_id = current_user`, so an invited member sees only their own (empty) data. This task creates a utility that resolves all user_ids sharing a workspace and modifies key queries to use `.in_("user_id", workspace_user_ids)` instead of `.eq("user_id", single_user_id)`.

**A. Create `app/services/workspace_data_filter.py`:**

A lightweight utility module with a single async function:

```python
"""Workspace-scoped data filter for shared content visibility.

When a user belongs to a workspace with multiple members, content queries
should include data from all workspace members — not just the requesting user.
This avoids adding workspace_id columns to every existing table.
"""

from __future__ import annotations

import logging
from functools import lru_cache

from app.services.workspace_service import WorkspaceService

logger = logging.getLogger(__name__)


async def get_workspace_user_ids(user_id: str) -> list[str]:
    """Return all user_ids sharing a workspace with the given user.

    If the user has no workspace or is the only member, returns [user_id]
    (single-element list). This means callers can always use `.in_()`
    instead of `.eq()` and get correct behavior for both solo and team users.

    Args:
        user_id: The requesting user's ID.

    Returns:
        List of user_id strings. Always contains at least the input user_id.
    """
    try:
        service = WorkspaceService()
        workspace = await service.get_workspace_for_user(user_id)
        if workspace is None:
            return [user_id]

        workspace_id = workspace["id"]
        members = await service.get_workspace_members(workspace_id)
        if not members:
            return [user_id]

        member_ids = [m["user_id"] for m in members]
        # Safety: ensure the requesting user is always included
        if user_id not in member_ids:
            member_ids.append(user_id)
        return member_ids
    except Exception:
        logger.warning("workspace_data_filter: falling back to solo user_id", exc_info=True)
        return [user_id]
```

Key design decisions:
- Always returns a list (never empty) so callers can uniformly use `.in_()`.
- Gracefully degrades to `[user_id]` on any error (workspace tables not yet migrated, service failure, etc.).
- Does NOT cache aggressively — workspace membership can change within a session. The two Supabase queries are fast enough for request-time resolution.
