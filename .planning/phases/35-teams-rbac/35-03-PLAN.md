**A. Create `frontend/src/components/ui/PermissionGate.tsx`:**

A reusable wrapper component that conditionally renders or disables children based on workspace role permissions.

```typescript
'use client';

interface PermissionGateProps {
  /** What permission level is required */
  require: 'edit' | 'manage-team';
  /** Content to render when permission is granted */
  children: React.ReactNode;
  /** Behavior when permission denied: 'hide' removes from DOM, 'disable' renders with disabled overlay (default) */
  fallback?: 'hide' | 'disable';
  /** Custom message shown when disabled (default: "Contact your workspace admin to perform this action") */
  deniedMessage?: string;
}
```

Logic:
- Use `useWorkspace()` to get `canEdit` and `canManageTeam`
- Map `require='edit'` to `canEdit`, `require='manage-team'` to `canManageTeam`
- If `ready` is false, render children normally (don't block during loading)
- If no workspace (solo user), render children normally (solo users have full access)
- If `fallback='hide'`, return null when denied
- If `fallback='disable'` (default), wrap children in a relative div with:
  - `pointer-events: none` and `opacity: 0.5` on the children
  - A tooltip/overlay that shows the denied message on hover
  - The denied message defaults to "Contact your workspace admin to perform this action"

**B. Create `frontend/src/components/team/RoleDropdown.tsx`:**

A select dropdown component for changing a member's role.

Props:
- `currentRole: string` — current role value
- `memberId: string` — the member's user_id
- `isOwner: boolean` — if true, dropdown is disabled (owner is always admin)
- `disabled?: boolean` — external disable (e.g., for non-admin viewers)
- `onRoleChange: (memberId: string, newRole: string) => Promise<void>` — callback

Renders a `<select>` with options: Admin, Editor, Viewer. Shows current role as selected. On change, calls onRoleChange. Shows loading state during the async call. If isOwner is true, renders "Owner (Admin)" as static text instead of a dropdown.

Style with Tailwind: rounded border, indigo focus ring, consistent with existing form elements in the codebase.

**A. Create `frontend/src/app/dashboard/team/page.tsx`:**

The team settings page at `/dashboard/team`. This is a `'use client'` page.

Structure:
1. Wrap in `<GatedPage featureKey="teams">` to enforce startup+ tier gating
2. Use `useWorkspace()` to get workspace state
3. If workspace is not ready, show loading shimmer
4. If no workspace exists yet, call the backend `GET /teams/workspace` to trigger lazy workspace creation (the backend's get_or_create_workspace creates one if none exists), then refresh the context
5. Page layout:
   - **Header section:** "Team Settings" heading with workspace name subtitle
   - **Members section:** `<TeamMemberList>` component showing all members
   - **Invite section (admin only):** `<InviteLinkGenerator>` wrapped in `<PermissionGate require="manage-team">` so only admins see the invite generation UI
   - **Role info section:** A card explaining the three roles:
     - Admin: Full access, can manage team members and billing
     - Editor: Can create and edit initiatives, workflows, and content
     - Viewer: Read-only access to all shared workspace content
6. If the user's role is 'viewer', the page still loads (viewers can see who's in their team) but the invite section is hidden and role dropdowns are disabled

Use PersonaDashboardLayout for consistent page structure (same as workspace/page.tsx pattern).

**B. Create `frontend/src/app/dashboard/team/join/page.tsx`:**

The invite acceptance page at `/dashboard/team/join?token=xxx`. This is a `'use client'` page.

Structure:
1. Read `token` from URL search params using `useSearchParams()`
2. Wrap in Suspense boundary (required for useSearchParams in Next.js)
3. States: `idle`, `loading`, `success`, `error`
4. On page load, show a card with:
   - "You've been invited to a workspace" heading
   - "Accept Invitation" button
5. On click, call backend `POST /teams/invites/accept` with `{ token }` and auth header
6. On success: show success message ("You've joined the workspace!"), call workspace context refresh, then redirect to `/dashboard/team` after 2 seconds
7. On error: show error message from backend (expired link, already a member, invalid token)
8. If no token in URL, show "Invalid invite link" error

The page does NOT need GatedPage wrapping — any authenticated user should be able to accept an invite (the backend validates tier requirements). However, the user must be logged in. If not authenticated, redirect to login with a return URL.
