Create a new Supabase migration file with three tables and their RLS policies:

**1. `workspaces` table:**
- `id` UUID PK default gen_random_uuid()
- `owner_id` UUID NOT NULL REFERENCES auth.users(id) — the user who created this workspace
- `name` TEXT NOT NULL — workspace display name (default to user's email prefix or "My Workspace")
- `slug` TEXT UNIQUE — URL-friendly identifier, auto-generated
- `created_at` TIMESTAMPTZ DEFAULT now()
- `updated_at` TIMESTAMPTZ DEFAULT now()
- RLS: owner can do everything; workspace members can SELECT
- Index on owner_id

**2. `workspace_members` table:**
- `id` UUID PK default gen_random_uuid()
- `workspace_id` UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE
- `user_id` UUID NOT NULL REFERENCES auth.users(id)
- `role` TEXT NOT NULL CHECK (role IN ('admin', 'editor', 'viewer')) — the three RBAC roles
- `joined_at` TIMESTAMPTZ DEFAULT now()
- UNIQUE(workspace_id, user_id) — one membership per workspace per user
- RLS: workspace members can SELECT their own workspace's members; admin can INSERT/UPDATE/DELETE
- Index on workspace_id, user_id

**3. `workspace_invites` table:**
- `id` UUID PK default gen_random_uuid()
- `workspace_id` UUID NOT NULL REFERENCES workspaces(id) ON DELETE CASCADE
- `token` TEXT NOT NULL UNIQUE — random invite token for share links
- `role` TEXT NOT NULL DEFAULT 'viewer' CHECK (role IN ('editor', 'viewer')) — role assigned on acceptance (admin cannot be invited, only the owner is admin initially)
- `created_by` UUID NOT NULL REFERENCES auth.users(id)
- `expires_at` TIMESTAMPTZ NOT NULL — invite expiry (default 7 days from creation)
- `accepted_by` UUID REFERENCES auth.users(id) — NULL until accepted
- `accepted_at` TIMESTAMPTZ — NULL until accepted
- `is_active` BOOLEAN DEFAULT true — can be revoked
- `created_at` TIMESTAMPTZ DEFAULT now()
- RLS: workspace admin members can CRUD; anyone can SELECT by token (for accepting)
- Index on token, workspace_id

**RLS Policy Strategy:**
- Use `service_role` for backend operations (the app uses service client for data access)
- For anon/authenticated direct access: members of a workspace can read workspace data; only admins can write membership records
- Add a helper function `is_workspace_member(ws_id UUID)` that returns TRUE if auth.uid() is in workspace_members for that workspace
- Add a helper function `get_workspace_role(ws_id UUID)` that returns the role text

**Important:** Do NOT add workspace_id columns to existing tables (initiatives, campaigns, etc.) in this migration. That is a larger data migration that would be done incrementally. For v5.0, workspace isolation works at the application layer — the workspace_members table tells us which users share data, and the backend service filters by user_ids that are in the same workspace.

**A. Create `app/services/workspace_service.py`:**

A service class `WorkspaceService` with these async methods:

1. `get_or_create_workspace(user_id: str) -> dict` — Returns the user's workspace. If none exists (user has no workspace_members row), creates a workspace with the user as owner + admin member. Returns workspace dict with id, name, slug, owner_id.

2. `get_workspace_for_user(user_id: str) -> dict | None` — Looks up workspace_members for user_id, returns the workspace record. Returns None if user has no workspace.

3. `get_member_role(user_id: str, workspace_id: str) -> str | None` — Returns the role string ('admin', 'editor', 'viewer') or None if not a member.

4. `get_workspace_members(workspace_id: str) -> list[dict]` — Returns all members of a workspace with their user profile info (email, name from user_profiles or auth.users, role, joined_at).

5. `create_invite_link(workspace_id: str, created_by: str, role: str = "viewer", expires_hours: int = 168) -> dict` — Creates a workspace_invites row with a secure random token (use `secrets.token_urlsafe(32)`), returns the invite record including the token. Validate that created_by is an admin of the workspace. Validate role is 'editor' or 'viewer' (not 'admin').

6. `accept_invite(token: str, user_id: str) -> dict` — Looks up the invite by token, validates it's active and not expired, checks the user isn't already a member, inserts into workspace_members with the invite's role, marks the invite as accepted. Returns the new membership record. Raise ValueError with descriptive message on any validation failure.

7. `update_member_role(workspace_id: str, target_user_id: str, new_role: str, actor_user_id: str) -> dict` — Changes a member's role. Only admins can do this. Cannot change the workspace owner's role (owner is always admin). Returns updated member record.

8. `remove_member(workspace_id: str, target_user_id: str, actor_user_id: str) -> bool` — Removes a member. Only admins can do this. Cannot remove the workspace owner. Returns True on success.

Use the existing `get_service_client()` and `execute_async()` patterns from `app/services/supabase.py` and `app/services/supabase_async.py`. Follow the docstring convention used throughout the codebase (Google-style docstrings with Args/Returns/Raises sections). Add `from __future__ import annotations` at top.

**B. Create `app/middleware/workspace_role.py`:**

A FastAPI dependency factory `require_role(*allowed_roles: str)` following the exact same pattern as `require_feature()` in `app/middleware/feature_gate.py`:

```python
def require_role(*allowed_roles: str) -> Callable[..., Any]:
    """Create a FastAPI dependency that gates an endpoint by workspace role.

    Usage:
        # Router-level (all endpoints require editor+)
        router = APIRouter(
            dependencies=[Depends(require_role("admin", "editor"))],
        )

        # Per-endpoint
        @router.delete("/{id}")
        async def delete_item(
            _role: None = Depends(require_role("admin")),
        ):
    """
```

The inner dependency should:
1. Get user_id via `Depends(get_current_user_id)`
2. Call `WorkspaceService().get_workspace_for_user(user_id)` to get workspace
3. If no workspace, return (no workspace = solo user, not blocked — team RBAC only applies to team members)
4. Call `WorkspaceService().get_member_role(user_id, workspace_id)`
5. If role not in allowed_roles, raise HTTPException(403) with structured JSON:
   ```json
   {
     "error": "insufficient_role",
     "message": "This action requires admin or editor role. Your role is viewer. Contact your workspace admin.",
     "current_role": "viewer",
     "required_roles": ["admin", "editor"]
   }
   ```

Also export a helper `get_workspace_context` dependency that resolves and returns the user's workspace_id and role as a dict (or None if no workspace). This is useful for endpoints that need workspace context but don't gate on role.
