<objective>
Migrate the four hot-path Supabase consumers (session service, task store, workflow engine, RAG) from sync-wrapped calls to native async calls using the new AsyncClient.

Purpose: These four subsystems handle every user request — sessions on every chat message, tasks on every A2A call, workflows on every workflow operation, RAG on every knowledge query. Moving them to native async eliminates ~90% of the thread pool pressure.

Output: All hot-path services use native async Supabase calls. No asyncio.to_thread for DB operations in these files.
</objective>

From a2a.server.tasks.task_store.TaskStore:
```python
class TaskStore:
    def get(self, task_id: str, context=None) -> Task | None: ...
    def save(self, task: Task, context=None) -> None: ...
    def delete(self, task_id: str, context=None) -> None: ...
```
Note: TaskStore interface is SYNC. The A2A library calls these synchronously.
For TaskStore, we need to use asyncio.run_coroutine_threadsafe or keep a
sync wrapper that calls the async client. The best approach: use the updated
execute_async's fallback — keep the sync TaskStore but ensure its .execute()
calls go through the async client's connection pool. Alternatively, wrap with
asyncio.get_event_loop().run_until_complete() since TaskStore methods are
called from within async context via A2A.

    The A2A `TaskStore` interface has SYNC method signatures (`def get`, `def save`, `def delete`). We cannot make these async. Two strategies:

    Strategy: Use `asyncio.get_event_loop().run_until_complete()` inside the sync methods to call the async client. However, this fails if an event loop is already running (which it always is in FastAPI). Instead, use the UPDATED `execute_async` fallback approach:

    Keep the sync `Client` reference for TaskStore specifically (it's a low-traffic path — A2A task operations are infrequent compared to sessions). But update it to use `get_service_client()` from supabase_client.py (which it already does via knowledge_vault re-export). Simply clean up the import path:
    - Replace `from app.rag.knowledge_vault import get_supabase_client` with `from app.services.supabase_client import get_service_client`

    This is acceptable because: (a) TaskStore is sync by interface contract, (b) A2A task operations are low-frequency (task create/update, not per-message), (c) The thread pool still handles these few sync calls fine even at reduced size.

    Add a docstring explaining WHY TaskStore remains sync: "A2A TaskStore interface mandates sync methods. These low-frequency operations remain on the sync client."
