<objective>
Harden the Supabase session service and rate limiting middleware for resilience when backend services degrade.

Purpose: The session service currently retries only network-level errors (ConnectError, ReadTimeout, WriteTimeout) but not HTTP 5xx responses from Supabase, and has no circuit breaker protection. The rate limiter silently bypasses all limits when Redis is unavailable. Both represent production availability gaps.

Output: Session service protected by the existing SupabaseCircuitBreaker with 5xx retry support; rate limiter falls back to in-process SlowAPI when Redis circuit breaker is open.
</objective>

**Change 2: Add a fallback check function**

```python
def _in_process_rate_check(
    user_id: str, limit: int, window_seconds: int = 60
) -> tuple[bool, int, int, int]:
    """In-process fixed-window rate check — fallback when Redis is unavailable.

    Less accurate than Redis sliding window (not shared across replicas),
    but ensures no single user can make unlimited calls.
    """
    now = int(time.time())
    window_start = (now // window_seconds) * window_seconds
    reset_at = window_start + window_seconds
    key = f"{user_id}:{window_start}"

    entry = _fallback_counters.get(key)
    if entry is None:
        # Clean expired entries (older than 2 windows) to prevent unbounded growth
        cutoff = window_start - window_seconds
        expired = [k for k in _fallback_counters if int(k.rsplit(":", 1)[-1]) < cutoff]
        for k in expired:
            del _fallback_counters[k]
        _fallback_counters[key] = (1, window_start)
        return True, limit, limit - 1, reset_at

    count, _ = entry
    count += 1
    _fallback_counters[key] = (count, window_start)
    remaining = max(0, limit - count)
    allowed = count <= limit
    return allowed, limit, remaining, reset_at
```
