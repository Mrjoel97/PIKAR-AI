<objective>
Add Supabase Realtime subscription to SubscriptionContext so the user's subscription badge updates in real-time when a Stripe webhook changes their row, without a page reload. Ship a reusable <SubscriptionBadge /> component that renders the current tier + billing state. BILL-03.

Purpose: The existing SubscriptionContext at frontend/src/contexts/SubscriptionContext.tsx loads the subscription row on mount and on auth state change, but does NOT subscribe to postgres_changes. After a successful Stripe checkout the user returns to /dashboard/billing via ?checkout=success which triggers a manual refresh(), but for lifecycle events (trial ended, payment failed, admin override) the UI stays stale until reload. The existing codebase already uses this exact realtime pattern in useRealtimeNotifications.ts, useRealtimeSession.ts, useRealtimeWorkflow.ts — mirror it here.

Output: Migration enabling Realtime replication for subscriptions, enhanced SubscriptionContext, SubscriptionBadge component, vitest coverage.

    ```sql
    -- Migration: 20260407100000_subscriptions_realtime.sql
    -- Description: Enable Supabase Realtime postgres_changes streaming for the
    -- subscriptions table so the frontend SubscriptionContext can subscribe to
    -- UPDATE events and reflect Stripe webhook-driven state changes in real time.
    --
    -- Without this, supabase.channel(...).on('postgres_changes', ...) will not
    -- deliver any events for the subscriptions table even though RLS grants
    -- the user SELECT on their own row.

    DO $$
    BEGIN
      IF NOT EXISTS (
        SELECT 1 FROM pg_publication_tables
        WHERE pubname = 'supabase_realtime'
          AND schemaname = 'public'
          AND tablename = 'subscriptions'
      ) THEN
        ALTER PUBLICATION supabase_realtime ADD TABLE subscriptions;
      END IF;
    END $$;
    ```

    Why a DO block (not plain ALTER): `ALTER PUBLICATION ... ADD TABLE` raises if the table is already in the publication, which breaks `supabase db reset --local` idempotency. The check-first pattern is the Supabase-documented safe form.
