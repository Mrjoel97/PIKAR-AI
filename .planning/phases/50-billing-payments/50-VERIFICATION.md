### Human Verification Required

See `human_verification:` block in frontmatter above. The 5 listed items are the Steps A-F from Plan 50-04 Task 4 that were explicitly deferred to a pre-beta smoke test because Docker Desktop, Stripe CLI, local Supabase, local backend, and local frontend were not all simultaneously available during the execution window. The deferred work is documented in Plan 50-04 SUMMARY under "Deferred to Pre-Beta Smoke Test".

**Why UAT is required (and not optional) for full phase closure:** The phase goal explicitly says "end-to-end through Stripe". While every individual layer is unit-tested and the Stripe-side contract is verified against real Stripe test mode (11/11 API assertions PASS), the integration path "real browser checkout → real Stripe webhook delivery → live Next.js handler → live Supabase UPSERT → live postgres_changes broadcast → open browser tab's badge rerender" is never observed as a single thread. Any one of those hops could be silently broken (e.g. wrong env var, wrong return URL, wrong redirect, wrong realtime publication config) and the unit tests would still pass. The deferred items are load-bearing.

**Why passing BILL-02 + BILL-04 without UAT is still honest:** BILL-02 is "webhook correctly processes events" — this is proven by the 9-case vitest suite + the Stripe API event payload shape verification, because both the webhook input contract (Stripe event shape) and the webhook output contract (Supabase upsert) are individually verifiable without a running stack. BILL-04 is "admin can view billing dashboard showing metrics" — the 14-case pytest suite covers the full router→service→DB path with mocked Supabase, and the metrics formula is pure Python arithmetic over mocked DB fixtures. No live stack is needed to prove either requirement is code-complete.

**Why the other 3 are human_needed:** BILL-01 ("user completes checkout and gets an active subscription"), BILL-03 ("real-time badge update without reload"), and BILL-05 ("manage subscription via Stripe Customer Portal") all depend on cross-process real-time state synchronization that no unit test can simulate. The code paths are all tested, but the integration is not observed.

### Gaps Summary

**Gap 1 — PremiumShell badge gating (minor, execution deviation):** The SubscriptionBadge is only rendered on routes that start with `/dashboard` because SubscriptionProvider is only mounted in `app/dashboard/layout.tsx`. Plan 50-04 truth #2 said "visible in the shell header on all authenticated pages for all personas" — the actual implementation covers dashboard routes (which include `/dashboard/billing` where checkout returns) but does NOT cover `/departments/*` or `/settings/*` pages that also use PremiumShell. This was discovered during Plan 50-04 execution and documented with an inline code comment. Two options to close: (a) lift SubscriptionProvider to a higher layout (e.g. root layout or an authenticated layout wrapping dashboard+departments+settings), or (b) accept the current scope as "badge visible where the user makes billing decisions" and update the truth. Neither option blocks the current phase goal, but this should be tracked as a Phase 52 or Phase 53 polish item.

**Gap 2 — Pre-beta smoke test deferred:** The full 6-step browser UAT (Steps A-F in Plan 50-04) is not yet observed. Every layer is independently tested and the Stripe-side contract is verified against real Stripe, but the end-to-end thread is not. This is tracked in Plan 50-04 SUMMARY as "Deferred to Pre-Beta Smoke Test" with exact run-command and prerequisites documented. The canonical helper `tests/e2e/test_stripe_checkout_flow.py` is ready to run when the full local stack is available.

**Gap 3 — Test 4 flakiness (infrastructure, not code):** `SubscriptionContext.test.tsx` Test 4 (`removes realtime channel on unmount`) was observed to fail once when run concurrently with `webhooks/stripe/__tests__/route.test.ts`. It passes reliably in isolation and in repeated solo runs. The implementation itself is correct — the `useEffect([userId])` cleanup function does call `supabase.removeChannel(channel)`. This is a vitest module-scope-state or timing artifact worth monitoring but not a code defect.

**Everything else is verified structurally by automated tests, and the Stripe-side contract is verified against real Stripe test infrastructure with 11/11 API assertions passing.
