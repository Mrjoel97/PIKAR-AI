<objective>
Harden the existing Next.js Stripe subscription webhook with an idempotency table, add the two missing lifecycle events (customer.subscription.created, invoice.payment_succeeded), AND close a latent event-ordering race by making customer.subscription.* events the sole source of truth for subscription state. Stripe retries, out-of-order deliveries, and past_due recoveries must update the user's subscription state correctly without manual intervention and without regressing fresh state. BILL-01 (event-ordering race) and BILL-02 (idempotency).

Purpose: The current handler at frontend/src/app/api/webhooks/stripe/route.ts processes 4 event types but has zero idempotency — a duplicate delivery (normal Stripe behaviour on retries) will overwrite fresh state with stale state if events arrive out of order. It also does not handle customer.subscription.created (only checkout.session.completed) and does not clear billing_issue_at after recovery.

CRITICAL event-ordering race to close (BILL-01): Stripe delivers customer.subscription.created BEFORE checkout.session.completed for new subscriptions, and retries checkout.session.completed independently. If both handlers upsert the same subscription row with tier/state, the following sequence loses user data:

  1. customer.subscription.created → writes row with tier=solopreneur, is_active=true, will_renew=true
  2. customer.subscription.updated → user toggles cancel_at_period_end=true → row shows will_renew=false
  3. checkout.session.completed (late retry, replayed after step 2) → overwrites row with stale tier/will_renew data from the original checkout session, REGRESSING the user's cancellation

The fix is to designate customer.subscription.created/updated/deleted as the SOLE source of truth for tier/is_active/will_renew/period/cancel_at_period_end, and demote checkout.session.completed to ONLY setting stripe_customer_id on the user's row (a pure customer-id → user-id mapping write). This matches Stripe's own recommended integration pattern (https://stripe.com/docs/billing/subscriptions/webhooks#events) where checkout.session.completed is treated as a fulfilment signal, not a state source.

Output: Migration for stripe_webhook_events, hardened webhook handler with demoted checkout handler, comprehensive vitest suite including an explicit regression test for the event-ordering race.
</objective>
