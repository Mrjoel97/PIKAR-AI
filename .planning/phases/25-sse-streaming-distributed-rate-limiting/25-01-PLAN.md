**Redis key design:**
- Per-user connection slot count: `pikar:sse:conn:{user_id}` (string, INCR/DECR with 5-minute TTL reset on each acquire — TTL covers process crash orphan cleanup, SSES-04)
- Per-user rate limiting key: `pikar:sse:rate:{user_id}` (string, INCR with 60s sliding window for RATE-02)
- SSE_MAX_CONNECTIONS_PER_USER env (default 3) — per-user slot limit (SSES-02)
- SSE_MAX_NEW_CONN_PER_MINUTE env (default 10) — per-user new connection rate per minute (RATE-02)
- SSE_MAX_TOTAL_CONNECTIONS env (default 500) — server-wide backpressure threshold (SSES-03)

**try_acquire_sse_connection logic:**
1. Get Redis client from `get_cache_service()._ensure_connection()`. If None (Redis down), log warning and return (True, 0, limit) — fail open, never block SSE due to Redis downtime.
2. Check server-wide backpressure: call `get_total_active_sse_count()`. If >= SSE_MAX_TOTAL_CONNECTIONS, log warning, return (False, current, limit). The returned tuple's `acquired=False` with `count >= limit` signals the CALLER to raise HTTP 503 (backpressure), not 429.
3. Check per-user rate limit: `INCR pikar:sse:rate:{user_id}`. If this is the first INCR (result == 1), set EXPIRE 60. If result > SSE_MAX_NEW_CONN_PER_MINUTE, return (False, current, limit). This signals the CALLER to raise HTTP 429.
4. Check per-user slot limit: `GET pikar:sse:conn:{user_id}`. If count >= per-user limit, return (False, count, limit). This signals the CALLER to raise HTTP 429.
5. `INCR pikar:sse:conn:{user_id}`. Set EXPIRE to SSE_CONN_TTL_SECONDS (default 300 = 5 min). Return (True, new_count, limit).

**Backpressure vs per-user discrimination — return a 4-tuple internally, expose a reason via a module-level enum or a separate helper:**

The cleanest approach is to have `try_acquire_sse_connection` return a named tuple or dataclass with a `reason` field so callers can distinguish:

```python
from enum import Enum

class SSERejectReason(str, Enum):
    PER_USER_LIMIT = "per_user_limit"
    PER_USER_RATE = "per_user_rate"
    SERVER_BACKPRESSURE = "server_backpressure"

# Internal dataclass (not exported in __all__, but callers can import it):
from dataclasses import dataclass

@dataclass
class SSEAcquireResult:
    acquired: bool
    active: int
    limit: int
    reason: SSERejectReason | None = None
```

Keep the existing 3-tuple return for backward compat AND add the dataclass version as the canonical return type. Callers updated in Task 2 will use the dataclass form.

Actually, to avoid a breaking change with any existing tuple-unpacking callers, use a subclass of tuple:

```python
class SSEAcquireResult(tuple):
    """Named result: (acquired: bool, active: int, limit: int). Has .reason attribute."""
    acquired: bool
    active: int
    limit: int
    reason: SSERejectReason | None

    def __new__(cls, acquired, active, limit, reason=None):
        instance = super().__new__(cls, (acquired, active, limit))
        instance.reason = reason
        return instance
```

Return `SSEAcquireResult(False, count, limit, reason=SSERejectReason.SERVER_BACKPRESSURE)` for server overload and `SSEAcquireResult(False, count, limit, reason=SSERejectReason.PER_USER_LIMIT)` for per-user limit. Existing tuple-unpack callers (`acquired, active, limit = await try_acquire(...)`) still work unchanged.

NOTE: Use `INCR` then `EXPIRE` (two commands) not a Lua script — simpler and the race window between them is acceptable for SSE slot management.

**release_sse_connection logic:**
1. Get Redis client. If None, log and return 0 (fail open).
2. `DECR pikar:sse:conn:{user_id}`. If result <= 0, `DEL pikar:sse:conn:{user_id}`, return 0.
3. Refresh TTL: `EXPIRE pikar:sse:conn:{user_id} {SSE_CONN_TTL_SECONDS}`.
4. Return remaining count.

**get_total_active_sse_count logic:**
1. Use `SCAN 0 MATCH pikar:sse:conn:* COUNT 1000` with async iteration.
2. For each key, `GET` the value and sum. Return total.
3. On Redis error, return 0 with warning log.

**reset_sse_connection_limits (test helper):**
