<objective>
Wire competitor monitoring alerts and build integration health monitoring.

Purpose: PROACT-03 requires users to get alerts when competitor changes are detected (pricing, launches, funding). The existing monitoring_job_service already runs research but doesn't dispatch user-facing alerts. PROACT-04 requires proactive warnings when OAuth tokens are expiring within 3 days or integrations are unhealthy. The existing integration_manager checks expiry at call time, not proactively. This plan:
1. Wires monitoring_job_service research results to the proactive alert dispatch system (from Plan 01)
2. Creates IntegrationHealthMonitor that proactively checks token expiry and connectivity
3. Adds Cloud Scheduler endpoints for both

Output: Competitor monitoring alerts dispatched + integration health alerts dispatched
</objective>

  <behavior>
    - Test: check_token_expiry finds tokens expiring within 3 days and returns list of (user_id, provider, expires_at, days_remaining)
    - Test: check_token_expiry ignores tokens with no expires_at (non-expiring API keys)
    - Test: check_token_expiry ignores tokens already expired (those should have been caught and refreshed by IntegrationManager)
    - Test: run_integration_health_check dispatches a WARNING notification for each expiring token with message like "Your Google Ads connection will expire in 2 days. Reconnect now to avoid disruption."
    - Test: run_integration_health_check deduplicates alerts (doesn't re-alert for same provider+user if already alerted today)
    - Test: check_connectivity pings provider health endpoints (or does a lightweight API call) and returns status
    - Test: run_integration_health_check dispatches an ERROR notification when connectivity check fails
  </behavior>
  <action>
    1. Create `app/services/integration_health_monitor.py`:
       - Class `IntegrationHealthMonitor` inheriting from `AdminService`.

       - Method `async check_token_expiry(days_threshold: int = 3) -> list[dict]`:
         - Query `integration_credentials` for all rows WHERE `expires_at` IS NOT NULL AND `expires_at` > now() AND `expires_at` < now() + interval '{days_threshold} days'.
         - For each row, compute days_remaining = (expires_at - now).days.
         - Return list of `{user_id, provider, account_name, expires_at, days_remaining}`.

       - Method `async check_connectivity() -> list[dict]`:
         - For each known provider in PROVIDER_REGISTRY that has a health_check_url or test endpoint:
           - Query `integration_credentials` for all active users of that provider.
           - For each user, attempt a lightweight API call (e.g., Google: userinfo endpoint, Stripe: /v1/balance, Slack: auth.test). Use httpx with 10s timeout.
           - If the call fails (timeout, 401, 403, 5xx): record {user_id, provider, status: "unhealthy", error}.
           - If the call succeeds: record {user_id, provider, status: "healthy"}.
         - Note: For MVP, only check providers that have simple health endpoints (Google, Slack, Stripe). Skip complex ones (Meta, HubSpot) that require specific scopes. Use a try/except approach -- if we can't check, mark as "unknown" (don't alert).
         - Return list of unhealthy results only.

       - Method `async run_integration_health_check() -> dict`:
         - Call `check_token_expiry()`. For each expiring token:
           - Format message: "Your {provider_display_name} connection will expire in {days_remaining} day(s). Please reconnect to avoid service disruption."
           - Determine link: "/settings/integrations" (the reconnect page).
           - Call `dispatch_proactive_alert(user_id, "integration.token_expiring", f"{provider}:{today}", title="Integration Expiring", message=..., notification_type=NotificationType.WARNING, link="/settings/integrations", metadata={provider, days_remaining, expires_at})`.
         - Call `check_connectivity()`. For each unhealthy result:
           - Format message: "Your {provider_display_name} connection is not responding. This may affect {affected_features}. Try reconnecting from Settings."
           - Call `dispatch_proactive_alert(user_id, "integration.unhealthy", f"{provider}:{today}", title="Integration Issue", message=..., notification_type=NotificationType.ERROR, link="/settings/integrations", metadata={provider, error})`.
         - Return {tokens_checked: N, tokens_expiring: N, connectivity_checked: N, unhealthy: N, alerts_sent: N}.

       - Use provider display name mapping: {"google": "Google Workspace", "google_ads": "Google Ads", "meta_ads": "Meta Ads", "slack": "Slack", "stripe": "Stripe", "hubspot": "HubSpot", "shopify": "Shopify"}.
       - Module-level convenience: `async def run_integration_health_check() -> dict`.

    2. Create `tests/unit/services/test_integration_health_monitor.py`:
       - Test check_token_expiry with mock credentials (some expiring, some not, some null).
       - Test check_connectivity with mock httpx responses (healthy and unhealthy).
       - Test run_integration_health_check dispatches correct alert types.
       - Test deduplication via dispatch_proactive_alert (mock it, verify alert_key includes today's date).
       - Mock all Supabase calls and httpx calls.
  </action>
