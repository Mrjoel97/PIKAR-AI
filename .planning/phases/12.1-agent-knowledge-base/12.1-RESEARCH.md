## Summary

The project already has a complete RAG stack: `app/rag/embedding_service.py` (Google GenAI `text-embedding-004`, 768-dim), `app/rag/ingestion_service.py` (chunking + batch embedding), `app/rag/search_service.py` (pgvector `match_embeddings` RPC), and `app/rag/knowledge_vault.py` (high-level API). The `embeddings` table (pgvector vector(768)) and `vault_documents` table already exist. Document parsing libraries `pypdf` and `python-docx` are already declared in `pyproject.toml`. Google Cloud Speech is also already installed.

Phase 12.1 builds an **admin-only knowledge management layer** on top of this existing RAG infrastructure. The core pattern is: (1) admin uploads file through AdminAgent chat → FastAPI multipart endpoint → Supabase Storage (`admin-knowledge` bucket) → content extracted by type → chunked/embedded → stored in `embeddings` table with `scope='system'` and `agent_scope=<agent_name_or_null>`. A new `admin_knowledge_entries` table tracks upload metadata and status. Retrieval is a filter-augmented call to the existing `match_embeddings` RPC.

The admin tool pattern is fully established: `_check_autonomy()` from `_autonomy.py`, `get_service_client()` for DB queries, `execute_async()` for async DB, and all tools registered in `admin_agent.py`. The REST router pattern is established in `config.py`. The frontend page pattern is established in `config/page.tsx`. Phase 12.1 follows all three patterns directly.

**Primary recommendation:** Reuse the existing `embeddings` table + `ingestion_service.py` + `embedding_service.py` — add only a new `admin_knowledge_entries` tracking table, a new `admin-knowledge` storage bucket, and a `scope` column to distinguish system-admin knowledge from user knowledge.

### Pattern 1: File Upload → Extract → Embed (Server-Side)

The admin never uploads files directly to Supabase Storage from the browser. The AdminAgent receives a chat message describing the upload, the frontend provides a multipart form upload to a FastAPI endpoint, the backend stores the file, extracts text, and embeds — all server-side.

**Why:** Admin panel is desktop-first. Multipart POST to FastAPI is simpler than presigned URL flows for the admin use case. The service-role key is already on the backend.

```python

    file_bytes: bytes,
    filename: str,
    agent_scope: str | None,  # None = global
    uploaded_by: str,         # admin user_id
) -> dict:
    # 1. Upload raw file to Supabase Storage bucket 'admin-knowledge'
    # 2. Extract text based on extension (PDF → pypdf, DOCX → python-docx, TXT/MD → raw)
```

### Pattern 3: AdminAgent Tool — Upload Knowledge

```python
# Source: established pattern from app/agents/admin/tools/config.py

async def upload_knowledge(
    filename: str,         # from chat context — admin names the file
    agent_scope: str | None = None,   # None = global
    confirmation_token: str | None = None,

    """Process and embed uploaded training document for an agent.

    if gate is not None and confirmation_token is None:
        return gate
    # ... delegate to knowledge_service.process_document()
```

**Note on file delivery:** The AdminAgent cannot receive binary file bytes — it is a text-based LLM. The flow is:
1. Admin types: "Train the financial agent with this quarterly report" and attaches a file in the chat UI
2. Frontend sends file via `POST /admin/knowledge/upload` (multipart) with `agent_scope=financial`
3. Backend processes and embeds immediately (no agent call needed for the heavy work)
4. AdminAgent `upload_knowledge` tool is called with the resulting `entry_id` to confirm and log the action

This is the correct design — the ADK agent handles orchestration and confirmation, not binary data.

### Anti-Patterns to Avoid

- **Uploading files through the AdminAgent LLM:** The agent cannot handle binary payloads. Files must go through a dedicated multipart REST endpoint. The agent handles intent + confirmation only.
- **Adding a new embeddings table for admin knowledge:** Use the existing `embeddings` table with `metadata->>'scope' = 'system'`. A second table would split the vector index and complicate search.
- **Modifying `match_embeddings` RPC signature:** Multiple callers depend on it. Add a new `match_system_knowledge` RPC instead.
- **Blocking on video processing in the HTTP request:** Video transcription can take 30+ seconds. Use `asyncio.to_thread()` for the audio extraction step and accept that the upload endpoint has a longer timeout — or use a background task pattern (ai_jobs table already exists for this).
- **User-scoped RLS on admin knowledge:** Admin knowledge is system-scoped. Access goes through the service-role client (bypasses RLS). Do not put admin-uploaded knowledge into the per-user `vault_documents` / `embeddings` rows.

## Common Pitfalls

### Pitfall 1: confusing `agent_id` (UUID FK) with `agent_name` (text) in the embeddings table

**What goes wrong:** The existing `embeddings` table has `agent_id UUID REFERENCES agents(id)` — a foreign key to the `agents` table which stores per-user agent instances. Admin knowledge is not associated with a per-user agent UUID. Using the UUID FK would require creating ghost agent rows.

**Why it happens:** The `embeddings.agent_id` column was designed for user-facing agent knowledge, not system-wide admin knowledge.

**How to avoid:** Store `agent_scope` in `metadata JSONB` as a plain text agent name (e.g., `metadata->>'agent_scope' = 'financial'`). Set `agent_id = NULL` on the `embeddings` row. The `admin_knowledge_entries` tracking table stores the agent name in a dedicated text column.

**Warning signs:** If you see a JOIN between `embeddings.agent_id` and the `agents` table for admin knowledge, it's wrong.

### Pitfall 2: Video processing timing out the HTTP request

**What goes wrong:** Uploading a 10-minute MP4, extracting audio, and transcribing via Speech-to-Text could take 60-120 seconds. FastAPI default request timeout (and Cloud Run) will kill the connection.

**How to avoid:** For videos, the upload endpoint should:
1. Store the raw file in Supabase Storage immediately
2. Insert an `admin_knowledge_entries` row with `status='processing'`
3. Enqueue an `ai_jobs` record (`job_type='admin_knowledge_video'`) for background processing
4. Return `202 Accepted` with the entry ID
5. A background worker picks it up via the existing `claim_next_ai_job` RPC

**Warning signs:** Any direct `await speech_client.recognize(video_bytes)` inside the upload handler.

### Pitfall 3: Stale instruction injection after knowledge is added

**What goes wrong:** The AdminAgent's instruction string (hardcoded in `ADMIN_AGENT_INSTRUCTION`) does not mention available knowledge tools. After adding Phase 12.1 tools, the agent won't know to use them unless the instruction is updated — same issue addressed in Phase 12 (Pitfall 1 from 12-RESEARCH.md).

**How to avoid:** Add a `## Knowledge Management Tools (Phase 12.1)` section to `ADMIN_AGENT_INSTRUCTION` in `agent.py`, following the exact pattern used for Phase 12 config tools in that file. The `_make_admin_runner()` async DB fetch ensures edits take effect without redeploy.

### Pitfall 4: `match_embeddings` RPC doesn't support `scope` filter

**What goes wrong:** The existing `match_embeddings` Postgres function filters by `filter_user_id` and `filter_agent_id`. There is no `scope` parameter. Calling it for system knowledge with `filter_user_id=NULL` would return ALL embeddings (user + system), mixing results.

**How to avoid:** Create a new `match_system_knowledge` RPC in the migration that filters on `metadata->>'scope' = 'system'` and optionally `metadata->>'agent_scope' IN (agent_name, NULL)`. Do not modify `match_embeddings`.

**Warning signs:** Any call to `supabase.rpc("match_embeddings", {...})` in the knowledge search path without a scope discriminator.

### Pitfall 5: Image "embedding" — embedding the raw image bytes

**What goes wrong:** The `text-embedding-004` model is text-only. You cannot embed image bytes.

**How to avoid:** For images, the workflow is:
1. Store image in Supabase Storage
2. Call Gemini vision (e.g., `gemini-2.5-flash` with image part) to generate a text description
3. Embed the text description + filename + user-provided metadata
4. Store the Storage URL in `admin_knowledge_entries.file_path` for reference

This is the correct pattern — the knowledge base stores semantic descriptions, not raw pixels.

## Open Questions

1. **Video processing: sync vs. background job**
   - What we know: Videos >30s will timeout synchronous HTTP. The `ai_jobs` table and `claim_next_ai_job` RPC exist for background work.
   - What's unclear: Is the Phase 12.1 scope narrow enough (short training videos only) to stay synchronous with a generous timeout, or must it use the `ai_jobs` pattern?
   - Recommendation: Default to background job pattern (`ai_jobs`) for video. Return `202 Accepted` immediately. Status polling via `GET /admin/knowledge/{entry_id}`. This is safe for all video sizes and matches the existing infrastructure.

2. **Agent retrieval integration: when do specialized agents search system knowledge?**
   - What we know: The existing user-facing agents (`app/agents/financial/`, etc.) have tool sets defined in their respective `agent.py` files. None currently call system knowledge search.
   - What's unclear: Phase 12.1 requirements do not explicitly require wiring system knowledge into user-facing agent queries — KNOW-06 says "available to all agents" but there is no requirement for an agent tool to auto-search.
   - Recommendation: Phase 12.1 delivers the infrastructure (storage + embeddings + admin management). Agent retrieval integration (injecting a `search_system_knowledge` tool into each specialized agent) is a separate concern that can be a Wave 0 task or deferred to a later phase. Confirm scope with planner.

3. **File size limits**
   - What we know: The existing `knowledge-vault` bucket has a 50MB limit. Admin knowledge bucket should have a similar or higher limit. Large PDFs (100+ pages) may generate 500+ chunks.
   - What's unclear: Is there a practical concern about embedding cost for very large documents?
   - Recommendation: Set admin-knowledge bucket limit to 100MB. Add a soft warning in the tool response when chunk_count > 200.
