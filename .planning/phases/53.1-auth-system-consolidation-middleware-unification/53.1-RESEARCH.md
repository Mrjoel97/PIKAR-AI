# Phase 53.1: Auth System Consolidation & Middleware Unification - Research

**Researched:** 2026-04-10
**Status:** Complete

## Research Question

What is the lowest-risk way to unify the existing auth, middleware, and invite trust paths without regressing the already-working v7 flows?

## Findings

### 1. Backend auth is already strongest in `app/app_utils/auth.py`

- `verify_token()` performs local JWT verification first, enriches cached user data, and falls back to `supabase.auth.get_user()` only on cache miss.
- The duplicated route-local auth helpers in onboarding and action-history are strictly weaker because they always call Supabase and do not share the cache/strict-mode path.
- The lowest-risk consolidation is to add a canonical FastAPI dependency in `app/app_utils/auth.py` and make onboarding re-export it for compatibility.

### 2. Middleware identity drift comes from request-state population, not the Redis limiter itself

- `redis_sliding_window_check()` is already keyed by `user_id` and is structurally fine.
- The gap is that `RateLimitHeaderMiddleware` depends on `request.state.user_id`, while `RequestLoggingMiddleware` currently populates that from `x-user-id` / `user-id` headers.
- The correct fix is a shared request-identity resolver that decodes the bearer token locally and only falls back to headers when no authenticated identity is present.

### 3. The hardened Next proxy helper is already available

- `frontend/src/lib/supabase/proxy.ts` uses `getClaims()`, which is the safer server-side trust path.
- The active proxy in `frontend/src/proxy.ts` duplicates session refresh logic with `getUser()`.
- The best path is to reuse the helper for auth validation, then keep the existing persona/onboarding redirect logic on top of it.

### 4. Return URL handling is already implemented in auth pages

- Login, signup, and OAuth callback already sanitize and respect `returnUrl`.
- The missing link is generic protected-route redirects in `frontend/src/proxy.ts`, which currently send unauthenticated users to `/auth/login` without preserving the path they attempted to access.
- The fix should build a sanitized `returnUrl` from the current pathname + search when redirecting protected routes.

### 5. Invite metadata should move to a public backend endpoint

- The current invite metadata route in the frontend app uses a service-role key to read `workspace_invites`, `workspaces`, and inviter profile data.
- This is server-only in practice, but the user explicitly wants that privilege boundary kept out of the frontend runtime.
- The clean architecture is a FastAPI public endpoint that returns only display-safe invite metadata using the backend’s service-role access path.
- The frontend invite page can call that endpoint directly using `NEXT_PUBLIC_API_URL` through the existing public API fetch helper.

## Recommended Phase Split

### Plan 01: Backend auth dependency + middleware identity

- Add canonical FastAPI dependency helpers in `app/app_utils/auth.py`
- Re-export or consume the shared helper in onboarding and action-history
- Resolve authenticated user identity from bearer tokens inside middleware
- Add Python regression tests around canonical auth resolution and rate-limit identity extraction

### Plan 02: Frontend proxy + invite privilege boundary cleanup

- Switch the active proxy to the hardened helper
- Preserve `returnUrl` for protected-route redirects
- Move invite metadata lookup into FastAPI
- Remove the frontend service-role invite-details route
- Delete the unused `ProtectedRoute` component
- Add regression coverage for invite metadata fetch behavior and/or proxy redirect sanitization

## Validation Architecture

This phase needs both backend and frontend validation:

- **Backend:** pytest coverage for canonical auth dependency behavior and request identity extraction
- **Frontend:** TypeScript + Vitest coverage for invite metadata/public fetch or proxy redirect sanitization
- **Manual:** browser invite acceptance pass after the backend public metadata endpoint replaces the frontend service-role route

## Risks and Pitfalls

### Pitfall 1: Breaking existing router imports

Many routers import `get_current_user_id` from onboarding. Replacing the onboarding implementation with a thin shared export avoids a risky mass import rewrite.

### Pitfall 2: Dropping refreshed cookies on redirects

Any redirect created after `updateSession()` must copy cookies from the refreshed response object. Otherwise silent refresh breaks on auth-required navigations.

### Pitfall 3: Reintroducing a server-secret dependency through a new Next route

Moving invite metadata to FastAPI only helps if the frontend stops calling a frontend service-role route afterward. The Next route should be removed or retired.

### Pitfall 4: Trusting spoofable user headers when Authorization exists

Middleware must prefer bearer-derived user identity over user-id headers. Header fallback is acceptable only when there is no authenticated bearer identity.

## Conclusion

The phase should be executed as two plans:

1. Backend trust-path consolidation
2. Frontend proxy and invite-boundary consolidation

That sequence minimizes regression risk because the backend becomes canonical before the frontend switches to it.
