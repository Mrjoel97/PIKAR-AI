# Phase 53.1: Auth System Consolidation & Middleware Unification - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Close the auth and middleware consistency gaps discovered immediately after Phase 53 so v7 onboarding and multi-user flows build on one trustworthy path:

1. **Canonical backend auth dependency** — route authentication must resolve through the shared auth utility in `app/app_utils/auth.py`, not route-local `get_current_user_id` variants.
2. **Middleware identity hardening** — request logging and rate limiting must derive user identity from authenticated bearer tokens, not from spoofable user-id headers.
3. **Next.js proxy consolidation** — the active `frontend/src/proxy.ts` flow must delegate auth validation to the hardened Supabase proxy helper and preserve sanitized `returnUrl` redirects for protected routes.
4. **Backend-owned privileged invite reads** — invite metadata lookup for `/invite/[token]` must live in FastAPI/server-only backend code rather than a frontend runtime route that expects service-role secrets.
5. **Legacy auth guard cleanup** — duplicate or unused client-side auth wrappers should be removed once server-side enforcement is authoritative.

**Out of scope for Phase 53.1**:
- Replacing Supabase auth with a different provider
- Redesigning onboarding or persona routing beyond what is required for auth correctness
- Reworking all admin auth middleware paths unless they are directly blocked by this phase
- Adding new invite product features beyond secure metadata lookup and acceptance continuity
- Load/performance work planned for Phase 55

</domain>

<decisions>
## Implementation Decisions

### Security Boundary

- **Do not expose sensitive Supabase secrets in the browser.** Only `NEXT_PUBLIC_*` values remain browser-visible. Service-role access stays in backend/server-only execution paths.
- **Backend owns privileged invite metadata reads.** The frontend should consume a public backend endpoint for invite metadata rather than opening `workspace_invites` with a service-role key inside the frontend app runtime.

### Canonical Auth Path

- **`app/app_utils/auth.py` is the single source of truth** for authenticated user resolution.
- **`app/routers/onboarding.py` becomes a compatibility shim** if needed, so existing imports continue to work while the implementation lives centrally.
- **No route-local JWT verification copies.** Any remaining duplicate auth helpers should import or depend on the shared utility.

### Middleware Identity

- **Bearer token first, headers second.** Middleware may still log fallback header-based identities for non-authenticated internal calls, but authenticated identity must come from the JWT path and must not be overridden by `x-user-id` / `user-id`.
- **Rate-limit enforcement should not depend on request headers.** The rate-limit middleware should resolve authenticated identity itself if upstream state is missing.

### Proxy and Redirects

- **Use the hardened Supabase proxy helper in `frontend/src/lib/supabase/proxy.ts`.**
- **Protected-route redirects must preserve sanitized `returnUrl`.** This includes generic redirects to `/auth/login`, not only the invite page.
- **Keep the current public/protected route split.** Public routes should remain fast-pathed without unnecessary Supabase work.

### Legacy Cleanup

- **Delete unused client-side auth wrappers** if search confirms they are not referenced.
- **Preserve existing UX.** Cleanup should not remove user-facing behavior already enforced by server layouts, proxy redirects, or backend dependencies.

### Claude's Discretion

The executor may decide without re-asking:
- Exact helper function names in `app/app_utils/auth.py`
- Whether the public invite metadata endpoint lives in `teams.py` or a sibling router, as long as it stays backend-owned
- Whether the compatibility auth export lives in onboarding router or another thin module
- Exact test file names and grouping, provided they verify the approved fixes

</decisions>

<specifics>
## Specific Ideas

- The best low-risk consolidation path is to centralize implementation first and preserve import compatibility second.
- Any redirect helper should reuse cookies from the Supabase refresh response so proxy-based token rotation is not broken by redirects.
- Invite metadata should return only display-safe fields: workspace name, inviter name, role, invited email, expiry, and active status.

</specifics>

<code_context>
## Existing Code Insights

### Backend

- `app/app_utils/auth.py` already verifies JWTs locally, caches validated users, and only calls `supabase.auth.get_user()` on cache miss.
- `app/routers/onboarding.py` and `app/routers/action_history.py` each define their own `get_current_user_id`.
- Many backend routers import `get_current_user_id` from `app/routers/onboarding.py`, so onboarding is currently acting as an informal auth dependency hub.
- `app/fast_api_app.py` has both `RateLimitHeaderMiddleware` and `RequestLoggingMiddleware`, but request-state user identity currently comes from `x-user-id` / `user-id` headers rather than bearer auth.

### Frontend

- `frontend/src/lib/supabase/proxy.ts` already implements the safer `getClaims()`-based proxy session update path.
- `frontend/src/proxy.ts` still reimplements auth using `supabase.auth.getUser()` and drops `returnUrl` on generic protected-route redirects.
- `frontend/src/services/auth.ts`, `frontend/src/app/auth/login/LoginPage.tsx`, `frontend/src/app/auth/signup/SignupPage.tsx`, and `frontend/src/app/auth/callback/route.ts` already support sanitized auth continuation.
- `frontend/src/app/api/teams/invite-details/route.ts` currently reads `workspace_invites` directly with `SUPABASE_SERVICE_ROLE_KEY`.
- `frontend/src/components/auth/ProtectedRoute.tsx` appears unused.

</code_context>

<deferred>
## Deferred Ideas

- Broader admin auth consolidation beyond the scope of route identity and proxy trust paths.
- Invite acceptance UX polish beyond correctness and security.

</deferred>
