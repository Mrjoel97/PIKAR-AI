# Phase 27.1 Plan 01: Input Sanitization & Security Headers Summary

**One-liner:** html.escape() applied to all user-supplied HTML interpolation in 4 backend files; SecurityHeadersMiddleware extended with CSP, Referrer-Policy, and X-XSS-Protection.

## What Was Built

Applied `html.escape()` to every location where user-supplied strings were interpolated directly into HTML templates across the FastAPI backend, and extended `SecurityHeadersMiddleware` with the three missing security response headers required by SECU-01, SECU-02, and SECU-05.

### Files Modified

**app/mcp/tools/form_handler.py**
- Added `import html` at top
- `send_email_notification`: escaped all dict keys, dict values, `form_id`, and `submission_id` before HTML interpolation in the Resend email template

**app/mcp/tools/landing_page.py**
- Added `import html` at top
- `generate_html`: escaped `title`, `headline`, `subheadline` via pre-computed `e_*` variables; escaped `cta_text` in the form button; escaped `f["type"]`, `f["name"]`, `f["placeholder"]` for each form field input tag

**app/agents/tools/integration_tools.py**
- Added `import html` at top
- `send_message`: `html.escape(message_body)` before wrapping in `<p>` tag
- `start_call`: `html.escape(participant)` and `html.escape(when_value)` before wrapping in `<p>` tags

**app/routers/webhooks.py**
- Added `import html` at top
- `_forward_email` HTML-body path: `html.escape(original_from)` and `html.escape(subject)` in the attribution header div
- `_forward_email` plain-text path: `html.escape(original_from)`, `html.escape(subject)`, and `html.escape(body_text or '(empty body)')` in the `<pre>` block
- `body_html` left unescaped — it is forwarded HTML from the original sender, not user text injected into a template

**app/middleware/security_headers.py**
- Extended `SECURITY_HEADERS` dict with:
  - `Content-Security-Policy`: default-src 'self'; script-src includes unsafe-inline/unsafe-eval for Next.js SSE compatibility; frame-ancestors 'none'
  - `Referrer-Policy`: strict-origin-when-cross-origin
  - `X-XSS-Protection`: 1; mode=block
- Updated docstring to reflect full header set

**tests/unit/test_input_sanitization.py** (created)
- 23 regression tests across 4 test classes
- `TestFormHandlerEscaping` (4 tests): key/value escaping + form_id/data values in email HTML
- `TestLandingPageEscaping` (7 tests): title, headline, subheadline, cta_text, form field name/placeholder/type
- `TestIntegrationToolsEscaping` (2 tests): send_message body, start_call participant
- `TestWebhookForwardEscaping` (3 tests): original_from/subject in plain-text path, original_from in HTML path
- `TestSecurityHeadersMiddleware` (7 tests): CSP present + default-src, Referrer-Policy present + value, X-XSS-Protection present + value, existing headers preserved

## Decisions Made

1. **stdlib html.escape() only** — Python's built-in `html.escape()` is sufficient for server-side template escaping; no new dependency required.

2. **CSP unsafe-inline/unsafe-eval** — Next.js SSE streaming and inline event handlers in generated landing pages require these directives at this stage. The plan explicitly notes this as a practical starting CSP to be tightened with nonces in a future hardening phase.

3. **body_html forwarded verbatim** — In `_forward_email`, when `body_html` is present it is the original sender's HTML content being forwarded wholesale, not user-supplied text being injected into a structural template. Escaping it would corrupt the forwarded email display. Only `original_from` and `subject` (structural metadata) are escaped in this path.
