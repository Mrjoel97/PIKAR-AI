Create `app/services/cohort_analysis_service.py` with class `CohortAnalysisService(BaseService)`:

Note: This service analyzes the user's OWN Stripe subscription customers. The `subscriptions` table in Pikar tracks the Pikar user's own subscription. For SaaS cohort analysis, we need to use `financial_records` (Stripe-synced transactions) grouped by month to simulate cohort behavior. The cohort is defined by the first transaction month (signup month proxy).

1. **compute_cohort_retention(user_id: str, months: int = 6) -> dict** — Compute monthly cohort retention:
   - Query `financial_records` WHERE `source_type = 'stripe'` AND `user_id = user_id` AND `transaction_type = 'revenue'`
   - Group transactions by `source_id` (customer identifier from Stripe balance transaction) to identify unique customers
   - For each customer, determine their "signup month" = month of their first transaction
   - For each subsequent month, check if the customer had at least one transaction
   - Build retention matrix: `{cohort_month: {month_0: 100%, month_1: X%, month_2: Y%, ...}}`
   - Return `{"cohorts": dict, "months_analyzed": int, "total_customers": int}`

2. **compute_ltv_by_cohort(user_id: str, months: int = 6) -> dict** — Compute average LTV per cohort:
   - Same customer identification as retention (first transaction month = cohort)
   - For each cohort, sum all revenue from customers in that cohort, divide by customer count
   - Return `{"cohorts": {month: {"avg_ltv": float, "total_revenue": float, "customer_count": int}}, "overall_avg_ltv": float}`

3. **compute_churn_by_cohort(user_id: str, months: int = 6) -> dict** — Compute churn rate per cohort:
   - A customer is "churned" if they had no revenue transactions in the most recent 30 days
   - Group by signup month cohort
   - Churn rate = churned_count / total_count per cohort
   - Return `{"cohorts": {month: {"churn_rate": float, "churned": int, "total": int}}, "overall_churn_rate": float}`

4. **full_cohort_analysis(user_id: str, months: int = 6) -> dict** — Combine all three analyses:
   - Call compute_cohort_retention, compute_ltv_by_cohort, compute_churn_by_cohort
   - Use Gemini Flash to generate a 3-4 sentence plain-English summary highlighting key findings (best/worst retention cohort, LTV trends, churn concerns)
   - Return combined dict with all three results plus `executive_summary` string and `chart_data` for each metric

For all DB queries, use `execute_async` wrapping Supabase client calls. Use `from app.services.supabase_async import execute_async`. Import `BaseService` from `app.services.base_service`. Lazy-import Gemini only when generating summary text.

Create tests following established pattern from `test_observability_metrics_service.py`:
- Windows-safe sys.modules stubs
- Mock Supabase with fake financial_records rows spanning 3 months with multiple source_ids (customers)
- Verify retention percentages with known data
- Verify LTV calculations with known revenue amounts
- Verify churn detection based on recency threshold
