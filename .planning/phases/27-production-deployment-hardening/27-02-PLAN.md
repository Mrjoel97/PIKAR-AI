**docker-compose.prod.yml:**
Create a production Docker Compose override file. This file is used with `docker compose -f docker-compose.yml -f docker-compose.prod.yml up`. It overrides only the backend service's command to remove the dev-mode uvicorn --reload and use the Dockerfile's built-in CMD (gunicorn). Also removes the volume mounts (no hot-reload in production) and sets ENVIRONMENT=production.

```yaml
# Production overrides for docker-compose.yml
# Usage: docker compose -f docker-compose.yml -f docker-compose.prod.yml up
services:
  backend:
    # Remove dev command — Dockerfile CMD uses gunicorn with gunicorn.conf.py
    command: []
    volumes: []
    environment:
      - ENVIRONMENT=production
      - REDIS_HOST=redis
      - REDIS_PORT=6379
      - HOME=/home/appuser
    develop: !reset null
```

NOTE: The `command: []` with empty array tells Docker Compose to NOT override the Dockerfile CMD, letting gunicorn run. If docker-compose v2 does not support `command: []` for "use Dockerfile CMD", use the explicit gunicorn command instead: `command: ["sh", "-c", "uv run gunicorn app.fast_api_app:app --config gunicorn.conf.py"]`. The `develop: !reset null` removes the watch config from the base file. If the YAML tag is not supported, simply omit it and leave a comment.

**app/middleware/rate_limiter.py:**
Replace the synchronous in-memory `_get_cached_persona` and `_set_cached_persona` functions with async Redis-backed versions that fall back to the existing in-memory dict when Redis is unavailable. The key change is that these functions become `async` — which requires updating `get_user_persona_limit` to handle the async calls properly (it's already sync, called by slowapi which doesn't support async key_func).

IMPORTANT: `get_user_persona_limit` is used as a slowapi dynamic limit function which must be synchronous. The Redis cache check cannot be awaited in this sync context. Solution: keep the sync in-memory dict as a LOCAL L1 cache, and add a separate async function `refresh_persona_cache(user_id)` that reads from Redis (L2) into the in-memory dict. The SSE endpoint or middleware calls `refresh_persona_cache` asynchronously. The sync `get_user_persona_limit` reads from the in-memory dict as before — but the dict is now populated from Redis rather than only from DB lookups.

Concrete changes:
1. Add Redis key prefix constant: `_REDIS_PERSONA_PREFIX = "pikar:persona:"`
2. Modify `_set_cached_persona` to also write to Redis (async version):
   ```python
   async def _set_cached_persona_async(user_id: str, persona: str) -> None:
       """Cache persona in Redis (shared across replicas) + local memory."""
       # Local L1 cache (same as before)
       _persona_cache[user_id] = (persona, time.time())
       # ... existing cleanup logic ...

       # Redis L2 cache (shared across replicas)
       try:
           from app.services.cache import get_cache_service
           redis = await get_cache_service()._get_redis()
           if redis:
               await redis.setex(
                   f"{_REDIS_PERSONA_PREFIX}{user_id}",
                   _cache_ttl_seconds,
                   persona,
               )
       except Exception:
           logger.debug("Redis persona cache write failed, local cache still active")
   ```

3. Add `_get_cached_persona_async` that checks Redis when local cache misses:
   ```python
   async def _get_cached_persona_async(user_id: str) -> str | None:
       """Get persona: L1 (local dict) then L2 (Redis)."""
       # L1: local in-memory check (fast, no await)
       local = _get_cached_persona(user_id)
       if local:
           return local

       # L2: Redis check (shared across replicas)
       try:
           from app.services.cache import get_cache_service
           redis = await get_cache_service()._get_redis()
           if redis:
               persona = await redis.get(f"{_REDIS_PERSONA_PREFIX}{user_id}")
               if persona:
                   persona_str = persona.decode() if isinstance(persona, bytes) else str(persona)
                   # Backfill L1 cache
                   _persona_cache[user_id] = (persona_str, time.time())
                   return persona_str
       except Exception:
           logger.debug("Redis persona cache read failed, falling back to local only")

       return None
   ```

4. Keep the existing sync `_get_cached_persona` and `_set_cached_persona` unchanged (they serve as the L1 cache layer and are used by the sync `get_user_persona_limit`).

5. Add a new async function `warm_persona_cache(user_id: str, persona: str)` that calls `_set_cached_persona_async`. This is called from the SSE endpoint or auth middleware when persona is resolved from the DB.

6. In `get_user_persona_limit`, after the existing DB lookup fallback section (around line 126-128 where it currently just returns DEFAULT_LIMIT), the function already returns DEFAULT_LIMIT for unknown users. This sync path is fine — Redis warming happens in the async layer. No change needed to the sync function.
