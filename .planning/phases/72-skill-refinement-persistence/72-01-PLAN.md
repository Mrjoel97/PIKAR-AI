<objective>
Create the `skill_versions` Supabase table and wire the SelfImprovementEngine to persist every skill refinement as a versioned row and support rollback to the previous version.

Purpose: Without persistence, refined skills evaporate on Cloud Run cold starts. Without revert, a bad refinement is irreversible. This plan lays the DB foundation and wires the two critical engine methods.

Output: Migration SQL, updated engine with transactional write-through and DB-backed revert, unit tests.
</objective>

```sql
CREATE TABLE IF NOT EXISTS skill_versions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    skill_name TEXT NOT NULL,
    version TEXT NOT NULL,
    knowledge TEXT,
    previous_version_id UUID REFERENCES skill_versions(id),
    source_action_id UUID REFERENCES improvement_actions(id),
    created_by TEXT NOT NULL DEFAULT 'system:self-improvement-engine',
    created_at TIMESTAMPTZ DEFAULT now() NOT NULL,
    is_active BOOLEAN DEFAULT FALSE NOT NULL,
    metadata JSONB DEFAULT '{}'
);
```

Add the unique partial index enforcing exactly one active version per skill:

```sql
CREATE UNIQUE INDEX idx_skill_versions_active
```

Add supporting indexes:

```sql
CREATE INDEX idx_skill_versions_name ON skill_versions(skill_name, created_at DESC);
CREATE INDEX idx_skill_versions_active_lookup ON skill_versions(skill_name, is_active);
```

Add RLS policies (service_role full access, authenticated read):

```sql
ALTER TABLE skill_versions ENABLE ROW LEVEL SECURITY;

CREATE POLICY skill_versions_write_access ON skill_versions
    FOR ALL USING (auth.role() = 'service_role');
```

<task type="auto">
  <name>Task 2: Wire write-through and revert into SelfImprovementEngine</name>
  <files>app/services/self_improvement_engine.py</files>
  <action>
Modify `_execute_skill_refined` in `app/services/self_improvement_engine.py`:

1. **Before updating in-memory skill**, query `skill_versions` for the current active row for this skill_name (if any). Store its `id` as `previous_version_id`.

2. **Deactivate the previous version**: If a previous active row exists, UPDATE `skill_versions SET is_active = false WHERE id = {previous_version_id}`.

3. **Insert the new version**: INSERT into `skill_versions` with:
   - `skill_name`: skill.name
   - `version`: the new bumped version string
   - `knowledge`: the improved_knowledge text
   - `previous_version_id`: the id from step 1 (or NULL for first refinement)
   - `source_action_id`: action.get("id")
   - `created_by`: `_SYSTEM_USER_ID`
   - `is_active`: True
   - `metadata`: `{"effectiveness_before": action.get("metadata", {}).get("effectiveness_score")}`

4. **Then update in-memory registry** (existing code: `skill.knowledge = improved_knowledge.strip()`, version bump, changelog).

5. Include the `new_version_id` (from the insert response) in the return dict.

Modify `_attempt_revert`:

1. For `action_type == "skill_refined"`, query `skill_versions` for the current active version of the skill:
   ```python
   active_resp = await execute_async(
       self.client.table("skill_versions")
           .select("id, previous_version_id")
           .eq("skill_name", skill_name)
           .eq("is_active", True)
           .limit(1),
       op_name="self_improvement.revert_find_active",
   )
   ```

2. If no active version or no `previous_version_id`, log a warning and return (no revert possible).

3. Fetch the previous version row by its `previous_version_id`:
   ```python
   prev_resp = await execute_async(
       self.client.table("skill_versions")
           .select("*")
           .eq("id", previous_version_id)
           .single(),
       op_name="self_improvement.revert_fetch_prev",
   )
   ```

4. Deactivate the current active version: `UPDATE skill_versions SET is_active = false WHERE id = current_id`.

5. Activate the previous version: `UPDATE skill_versions SET is_active = true WHERE id = previous_version_id`.

6. Restore in-memory registry: `skill.knowledge = prev_data["knowledge"]` and `skill.version = prev_data["version"]`.

7. Log the revert: `logger.info("Reverted skill '%s' from v%s to v%s", ...)`.

Important: Use try/except around all DB operations so a DB failure does not crash the engine. Log warnings on failure. The in-memory update should still happen as a best-effort fallback even if DB write fails (matching existing pattern).
