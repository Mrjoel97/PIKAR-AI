  <action>
Create `frontend/src/config/featureGating.ts` as the SINGLE SOURCE OF TRUTH for all feature gating decisions. This file must be the only place that defines which persona tiers can access which features.

Define the following:

1. `PersonaTier` type — reuse from onboarding.ts: `'solopreneur' | 'startup' | 'sme' | 'enterprise'`

2. `TIER_ORDER` constant — ordered array `['solopreneur', 'startup', 'sme', 'enterprise']` (ascending access level). Used to determine "minimum tier required".

3. `FeatureKey` type — string union of all gated feature identifiers. Use route-path-based keys that map to sidebar nav hrefs. Keys:
   - `'workflows'` — Workflow Engine (startup+)
   - `'sales'` — Sales Pipeline & CRM (startup+)
   - `'compliance'` — Compliance Suite (sme+)
   - `'finance-forecasting'` — Financial Forecasting (sme+)
   - `'custom-workflows'` — Custom Workflows (enterprise only)
   - `'governance'` — SSO & Governance (enterprise only)
   - `'reports'` — Reports (startup+)
   - `'approvals'` — Approvals (startup+)

4. `FEATURE_ACCESS` record — `Record<FeatureKey, { label: string; description: string; minTier: PersonaTier; route?: string }>`. The `minTier` field specifies the lowest tier that can access the feature. The `route` field maps to the sidebar/page route (e.g., `'/dashboard/workflows'`). The `label` is the user-facing feature name. The `description` is a short one-line explanation shown in the upgrade prompt.

5. `isFeatureAllowed(featureKey: FeatureKey, userTier: PersonaTier): boolean` — returns true if the user's tier is >= the feature's minTier based on TIER_ORDER index comparison.

6. `getRequiredTier(featureKey: FeatureKey): PersonaTier` — returns the minTier for a feature.

7. `getFeatureKeyForRoute(route: string): FeatureKey | null` — given a route path, returns the matching FeatureKey or null if the route is ungated.

8. `getGatedRoutes(): Map<string, FeatureKey>` — returns a map of route -> FeatureKey for all gated features that have a route.

IMPORTANT: The existing FEATURE_ROWS on the billing page must stay consistent with this config. The billing page should eventually import from this config (but do NOT modify the billing page in this task — that can be a follow-up). The access matrix MUST match:
- solopreneur: AI Agents, Brain Dump, Invoice Gen, Social Publishing (all ungated base features)
- startup: + Workflow Engine, Sales Pipeline, Reports, Approvals
- sme: + Compliance Suite, Financial Forecasting
- enterprise: + Custom Workflows, SSO & Governance

Export all types and functions. Add JSDoc comments. Use the copyright header consistent with the project: `// Copyright (c) 2024-2026 Pikar AI. All rights reserved.`

**Create `frontend/src/hooks/useFeatureGate.ts`:**

A React hook that combines the user's current persona (from `usePersona()` context) with the gating config:

```typescript
export function useFeatureGate(featureKey: FeatureKey): {
  allowed: boolean;      // true if user's tier can access this feature
  currentTier: PersonaTier | null;  // user's current tier (null if loading)
  requiredTier: PersonaTier;        // minimum tier needed
  isLoading: boolean;    // true while persona is loading
  featureLabel: string;  // human-readable feature name
}
```

Implementation:
- Import `usePersona` from `@/contexts/PersonaContext`
- Import `isFeatureAllowed`, `getRequiredTier`, `FEATURE_ACCESS` from `@/config/featureGating`
- If persona is null (loading), return `{ allowed: false, currentTier: null, requiredTier, isLoading: true, featureLabel }`
- Otherwise compute allowed via `isFeatureAllowed(featureKey, persona)`

**Create `frontend/src/components/ui/UpgradePrompt.tsx`:**

A 'use client' component that renders a visually prominent upgrade prompt. Props:

```typescript
interface UpgradePromptProps {
  featureKey: FeatureKey;           // which feature is locked
  variant?: 'page' | 'sidebar' | 'card';  // rendering context
  className?: string;
}
```

The component must:
1. Call `useFeatureGate(featureKey)` internally to get tier info
2. Show the user's current tier name (e.g., "Solopreneur")
3. Show the locked feature name (from FEATURE_ACCESS[featureKey].label)
4. Show the feature description (from FEATURE_ACCESS[featureKey].description)
5. Show the required tier name (e.g., "Startup")
6. Include a CTA button linking to `/dashboard/billing` with text "View Plans"
7. Use Lucide `Lock` icon

**Variant rendering:**
- `'page'` (default): Full-page centered card with generous padding (p-8+), prominent heading, description text, and large CTA button. Shown when navigating to a gated page.
- `'sidebar'`: Compact inline banner (small text, horizontal layout) shown next to locked sidebar items.
- `'card'`: Medium card format for widget tiles or dashboard sections.

**Styling:** Use Tailwind classes consistent with the codebase:
- Background: `bg-gradient-to-br from-slate-50 to-slate-100` with `border border-slate-200`
- Lock icon: `text-slate-400`
- Feature name: `text-lg font-semibold text-slate-800`
- CTA button: `bg-indigo-600 hover:bg-indigo-700 text-white rounded-lg px-6 py-3`
- Show current tier with the tier's gradient color from PERSONA_INFO in onboarding.ts

If `isLoading` is true, render a shimmer placeholder (same pattern used in PersonaDashboardLayout loading state).

IMPORTANT: This is SOFT gating only — the CTA says "View Plans", NOT "Subscribe Now" or "Pay to Unlock". No Stripe integration. The billing page already shows the plan comparison.
