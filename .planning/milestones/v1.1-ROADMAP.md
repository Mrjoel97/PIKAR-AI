# Milestone v1.1: Production Readiness

**Status:** SHIPPED 2026-03-13
**Phases:** 2-6
**Total Plans:** 8

## Overview

v1.1 closed the gap between the running product and production-ready expectations. The milestone aligned the Supabase schema with the active runtime contract, removed blocking async data paths, repaired frontend/backend contract mismatches, hardened security controls, and finished the deployment/configuration guardrails needed to ship cleanly.

## Phases

### Phase 2: Database Alignment

**Goal:** The Supabase schema and codebase are in full agreement - no missing tables, no missing columns, no stale migrations
**Depends on:** Phase 1
**Plans:** 1 plan

Plans:

- [x] 02-01: Align skills/custom-skills schema, add content audit fields, and remove stale Alembic surfaces

**Details:**
Supabase SQL migrations became the only schema authority in the repo. The shipped work normalized `skills.agent_ids` to `jsonb`, aligned `custom_skills` to the live runtime contract, added missing ownership fields to content/workspace tables, removed the stale Alembic/SQLAlchemy migration path, and directly verified the resulting schema in the local database after working around the lagging local storage stack.

### Phase 3: Async Safety

**Goal:** No service method blocks the Python event loop - all database and cache calls are async-safe
**Depends on:** Phase 2
**Plans:** 1 plan

Plans:

- [x] 03-01: Convert blocking Supabase and Redis paths to async-safe execution, including cache initialization hardening

**Details:**
The affected service layer now routes Supabase access through async-safe execution helpers, `DatabaseSkillLoader` uses `CacheService` public APIs instead of private Redis access, and cache initialization/reset logic tolerates concurrent async callers. Verification included targeted async/cache tests plus an audit of remaining `.execute()` call sites.

### Phase 4: Frontend-Backend Alignment

**Goal:** The frontend communicates with the backend without auth failures, CORS errors, or type mismatches
**Depends on:** Phase 3
**Plans:** 2 plans

Plans:

- [x] 04-01: Route protected frontend pages through authenticated helpers and align initiative detail with the backend API contract
- [x] 04-02: Align workflow TypeScript contracts and add SSE heartbeat/keepalive hardening

**Details:**
Protected frontend pages now use shared authenticated service helpers, the initiative detail page consumes the backend's normalized contract, workflow TypeScript models match the API responses, and SSE streams gained keepalive behavior suitable for Cloud Run/proxy timeouts. The approval flow was clarified rather than forced into one model: the token link remains public while authenticated approval actions stay on authenticated helper paths.

### Phase 5: Security Hardening

**Goal:** The application enforces security controls in production - no exposed tokens, no bypassable auth, no missing headers
**Depends on:** Phase 4
**Plans:** 2 plans

Plans:

- [x] 05-01: Add security headers middleware, reject wildcard CORS in production, and tighten auth/rate-limiter token handling
- [x] 05-02: Add file upload size guardrails and regression coverage

**Details:**
Responses now include baseline security headers, production startup fails fast on wildcard CORS, the rate limiter no longer falls back to unverified JWT decoding, auth logging no longer exposes token fragments, and uploads are rejected with a 413 when they exceed the configured size limit. The milestone also added regression coverage around these production guardrails.

### Phase 6: Configuration and Deployment

**Goal:** The system deploys cleanly with a unified config, correct Docker health signaling, and no dev bypass flags in production
**Depends on:** Phase 5
**Plans:** 2 plans

Plans:

- [x] 06-01: Remove the dead AppSettings config surface, document shared guardrail envs, and add the backend Docker healthcheck
- [x] 06-02: Add shared per-user SSE connection limits and deployment guard regressions

**Details:**
The dead `AppSettings` and `CACHE_*` config surface was removed, Docker gained a first-class backend healthcheck, SSE admission now enforces per-user active stream limits with 429 responses, and production Terraform guardrails remain covered by regression tests. Local validation also exposed a Supabase CLI timing nuance: the CLI can return before `supabase_storage_Pikar-Ai` settles, even though the buckets and policies are already correct once the container finishes its post-reset health check.

## Milestone Summary

**Decimal Phases:**

- None

**Key Decisions:**

- Supabase SQL migrations are the only schema authority; the stale Alembic/ORM migration path is retired.
- Async request paths must use async-safe execution helpers and public cache interfaces only.
- The approval surface keeps the public token page public while authenticated approval/dashboard actions stay behind authenticated helpers.
- Production guardrails should fail fast rather than silently downgrade configuration safety.

**Issues Resolved:**

- Schema drift between the codebase and Supabase
- Blocking database/cache paths inside async request handling
- Frontend/backend auth, type, and SSE contract mismatches
- Missing security headers, permissive production CORS fallback, and unbounded upload size
- Dead config surfaces and missing deployment health signaling

**Issues Deferred:**

- No formal `.planning/v1.1-MILESTONE-AUDIT.md` artifact existed at archival time; closeout proceeded from completed phase summaries plus fully checked requirements.
- Local Supabase version-mismatch warnings for `gotrue` and `storage-api` still need a later `supabase link`.
- `uv.lock` still needs to be refreshed in an environment with full `uv lock` support.

**Technical Debt Incurred:**

- Local reset verification still relies on a short post-reset wait/health poll because `supabase_storage_Pikar-Ai` can settle after the CLI exits.
- A few unrelated `datetime.utcnow()` deprecation warnings remain outside the v1.1 scope.
