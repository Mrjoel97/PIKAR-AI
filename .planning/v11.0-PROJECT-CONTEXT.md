# v11.0 App Builder Beta — Project Context

**Started:** 2026-05-01
**Status:** In progress
**Decision source:** Direct user product decision after architectural audit (see commit history; chat log "go with recommended")

## Milestone Goal

Take the existing app-builder generation engine (shipped in v2.0, refined since) and finish the wrap so non-technical users can create end-to-end **landing pages and multi-page brochure websites** — and **publish them to a hosted URL** — without leaving the platform.

## What's IN scope

- Single-page landing pages
- Multi-page brochure websites (5–10 pages: home, about, services, portfolio, contact, etc.)
- Form submissions captured to a Supabase table + email notification
- Hosted publishing at `pikar.app/sites/{slug}` (static hosting via Supabase Storage + Cloud Run route)
- "My apps" dashboard for project management (resume, re-download, delete)
- Onboarding empty state + sample template seed
- Closed beta cohort: 50–100 invited users after Phase 94 ships

## What's OUT of scope (deferred)

| Feature | Deferred to | Why |
|---------|-------------|-----|
| Web app generation (backend logic, DB, auth for the user's app) | v12.0+ | Stitch generates frontend HTML only; backend generation is a separate engineering problem |
| Mobile app generation (real APK/IPA) | v12.0+ | Capacitor scaffold ZIP exists today but produces a developer setup, not an installable app; needs build infra (signing, EAS-style) |
| Custom domain mapping (`mycompany.com` → user's site) | v11.1 | Cloudflare wildcard at `*.pikar.app` covers beta; custom domains add DNS verification + cert provisioning complexity |

| Multi-user collaboration on the same project | v12.0+ | Single-owner is fine for beta; collab needs presence, role permissions, conflict resolution |
| Blog / CMS (markdown editor, posts table) | v11.1 or v12.0 | Adds 5–8 days; not needed for "landing + brochure" beta |
| E-commerce (Stripe checkout, product catalog) | v12.0+ | Adds 10–15 days; not needed for beta |
| Vercel API deploy as alternate target | v11.1+ | External dependency + per-user quotas; static hosting is simpler and fully under our control for beta |
| Real-time analytics on published sites | v11.1+ | Beta users won't push enough traffic to need it; can ship without |

## Locked Architecture Decisions

### Hosting path: **Supabase Storage + Cloud Run route**

When a user "ships," the final HTML/CSS/JS bundle is written to a public Supabase Storage bucket (`published_sites/{slug}/`). A Cloud Run route at `pikar.app/sites/{slug}` serves the HTML by reading from storage. HTTPS is provided by Cloudflare wildcard on `*.pikar.app`.

**Why not Vercel API:** External dependency, per-user pricing complexity, less control over caching strategy. Static hosting is simpler for beta and fully under our infra.
**Why not pure Cloudflare Pages:** Adds build pipeline complexity and external API surface. Cloud Run already handles every other request.
**Risk:** Subdomain routing edge cases, HTTPS cert provisioning, caching invalidation when users re-ship. Budgeted upper end of Phase 92 estimate (6 days) to absorb this. Cloudflare Workers is the fallback if Cloud Run static-serving gets gnarly.

### Beta scope: **Tier 1 only**

Tier 1 = single landing page + multi-page brochure (5–10 pages). The generation engine already handles this end-to-end (Stitch + multi-page builder + iteration loop + version history). No new generation tech needed for v11.0; the work is plumbing + product packaging.

**Why not Tier 2 (+ blog):** Adds 5–8 days; out of scope for closed beta. Beta users have not signaled blog as the gap.
**Why not Tier 3 (+ ecom):** Adds 10–15 days; blocks beta. Stripe integration + product catalog are their own milestone.

### Form submissions: **Supabase table + email notification**

`landing_form_submissions` table with columns: `id`, `slug`, `payload` (JSONB), `submitted_at`, `submitter_ip`, `seen` (bool). Project owner gets email via existing email integration. CSV export available from the project dashboard.

**Why not external form services (Formspree, Netlify Forms):** External dependency, per-submission pricing, harder to surface the data inside Pikar. Owning the storage layer keeps the data + UX cohesive.

### Slug strategy: **User-editable post-ship, with collision detection**

Default slug: `{first 8 chars of project_id}` (e.g. `abc12345`). User can rename to a friendly slug (e.g. `joes-coffee`) via the dashboard. Collision detection across all users; renames update the existing `published_sites` row, no orphan slugs.

**Why not pure UUID:** Friendly URLs are part of the share-with-clients value prop. URLs like `pikar.app/sites/joes-coffee` look credible; `pikar.app/sites/3f8a-91b2` looks broken.

## Beta Cohort Strategy

**Closed beta open to 50–100 invited users after Phase 94 ships.** Public beta is v12.0 — gated on collaboration, custom domains, and a real billing model. The closed beta cohort is for friction discovery + product-market validation, not scale.

**Invitation criteria:**
- Friendly enough to forgive bugs and provide feedback
- Non-technical enough to surface real onboarding pain (NOT engineers who can work around bugs)
- Spread across persona segments (solopreneur, SME, freelancer, agency)
- Willing to publish ≥1 landing page within 2 weeks of access

## Success Metric for v11.0 Closeout

**≥3 of the first 5 UAT users publish a working landing page at a `pikar.app/sites/*` URL within 60 minutes of receiving access.** This is the qualitative bar that says "the feature is real" before opening to a wider cohort.
