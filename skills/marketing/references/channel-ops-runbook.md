# Channel Ops Runbook

Use this reference when the user wants Codex to help operate live marketing channels in a semi-automated, approval-first way.

## Research Prerequisite

Before queueing, scheduling, or monitoring a channel, confirm there is a fresh research pack.

That pack should include:
- market research brief
- competitor teardowns
- channel benchmark
- viral framework library
- approved marketing brief

Do not operate channels blindly.

## Modes

### Draft Only

Use when access is limited or the user wants to review everything manually.

Output:
- approval queue
- publish-ready copy
- asset briefs
- reply drafts
- short reporting template

### Draft And Queue

Use when the user wants Codex to prepare the scheduling workload but keep publishing approval with a human.

Output:
- approval queue with dates and owners
- scheduler-ready entries
- platform notes
- status log

### Draft, Schedule, And Monitor

Use when the user wants Codex to help with execution inside an approved scheduler or dashboard.

Output:
- scheduled item log
- monitoring checklist
- metrics snapshot
- escalation notes

## Approval Gates

Always require explicit approval before:
- publishing a post
- sending an email or newsletter
- replying from a brand account
- changing ad spend or targeting
- editing bios, profile copy, or channel settings

Draft the action and ask for sign-off instead.

## Minimum Queue Schema

| Status | Channel | Date / Window | Format | Copy | Asset | CTA | URL | Source Pattern | Test Variable | Owner | Notes |
|--------|---------|---------------|--------|------|-------|-----|-----|----------------|---------------|-------|-------|

Recommended statuses:
- draft
- needs approval
- approved to schedule
- scheduled
- live
- blocked

## Monitoring Checklist

Review:
- comments and mentions
- DMs and warm leads
- engagement rate
- click-through rate
- follower or subscriber growth
- signups, demos, or conversions
- broken links and landing-page issues
- competitor activity if relevant

## Testing Rule

Run A/B tests from proven baselines. For each live test, note:
- the source pattern
- the one variable being changed
- the success metric
